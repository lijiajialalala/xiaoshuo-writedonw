# 小说内容内核发现系统操作方案

## 0. 一句话模型

```text
对标书不是目标，而是显微镜。
clean-room 开局不是生产，而是诊断工具。
真正目标是：从多本样本中发掘可迁移的小说内容内核，把内核适配到目标平台和题材，再用我们自己的 Story Bible / Style Bible 生产。
```

完整流程：

```text
平台 / 赛道研究
-> 样本池入库
-> 合规与相似风险边界
-> 前 10 章证据快拆
-> 路线候选判断
-> source-open 多颗粒度深读
-> 单书生成模型
-> route 横向归纳
-> 受控 clean-room 开局诊断
-> 盲审 / 追读诊断
-> 叙事身体抽取
-> 内容内核卡
-> 平台 / 赛道适配
-> 自有 Story Bible / Style Bible
-> 独立生产
-> 评估与回写
```

这套系统要同时保留两件事：

```text
旧模式的锋利反馈：clean-room + 盲审作为诊断显微镜。
新模式的安全边界：不把某本书、某作者、某个爆款开局当生产目标。
```

## 0.1 Skill 使用地图

当前文档不能只写流程，还要写每一步该用什么 skill / 审查模式。

本机当前未找到 `critical-thinking-mode` 这个 skill 文件；因此它先作为“必须补齐或以人工审查方式执行”的审查模式写入。后续如果安装了正式 skill，再把本节中的人工审查问题迁移进 skill。

### 0.1.1 必须使用的能力

| 能力/skill | 用在什么阶段 | 作用 | 当前状态 |
| --- | --- | --- | --- |
| `critical-thinking-mode` | A.1-A.15 全流程关键门 | 自问自答、反证、边界审查、禁止过早归纳 | 本机未找到，暂以人工审查清单执行 |
| `content-engine` | 内容内核卡、平台/赛道适配、house style、生产稿 | 把机制变成平台原生内容和可发布文本 | 可用 |
| `author-distiller` 等价能力 | 样本研究、source-open、route 归纳、runtime skill 草案 | 从样本中抽证据、归纳生成机制 | 小说侧暂未独立成 skill，先按本文档执行 |
| `blind-style-forensic-review` 等价能力 | clean-room 后的盲审/相似风险诊断 | 判断追读、自嗨、AI味、贴皮风险 | 小说侧暂未独立成 skill，先按盲审问题执行 |
| 平台研究能力 | P0.6 三平台机制研究 | 查官方规则、作者反馈、收益、合同和风控 | 已在平台研究文档执行 |

| 阶段 | 主 skill / 能力 | 必须问的问题 |
| --- | --- | --- |
| A.1 平台机制研究 | 平台研究 + `critical-thinking-mode` | 这是官方规则、作者反馈，还是媒体线索？它会改变我们生产什么文本吗？ |
| A.2 样本池入库 | `critical-thinking-mode` | 样本是不是偏向我们喜欢的书？有没有反例和普通样本？ |

| A.4 前10章快拆 | author-distiller 等价能力 | 证据能不能定位到章节？有没有把读后感当证据？ |
| A.5 route_candidate | `critical-thinking-mode` | 这是题材标签，还是生成路线？它能解释主角怎么赢吗？ |
| A.6 source-open 深读 | author-distiller 等价能力 | 表格有没有漏掉整段运动？哪些章节看似冗余但实际保追读？ |

| A.10 runtime skill 草案 | author-distiller 等价能力 | 每条规则有没有证据？有没有偷渡单书特色？ |
| A.11 clean-room 开局诊断 | `content-engine` | 生成稿能不能追读？有没有照搬原书内容形状？ |
| A.12 盲审/追读诊断 | blind-review 等价能力 + `critical-thinking-mode` | 反馈能不能定位问题？是不是只收到了主观好恶？ |
| A.13 叙事身体回修 | `critical-thinking-mode` | 失败是 route、因果、阻力、翻牌、文风还是相似风险？ |
| A.14 house style | `content-engine` + `critical-thinking-mode` | 这是我们的声音，还是“不像别人”的空壳？ |
| A.15 独立生产 | `content-engine` + `critical-thinking-mode` | 是否通过追读、相似风险、平台适配和质量门？ |

### 0.1.3 critical-thinking-mode 的人工执行规则

在正式 skill 补齐前，每个关键节点必须人工执行以下自问自答：

```text
我现在的结论依赖什么证据？
有没有反例？
有没有把单本书特色当成 route 规律？
有没有把题材标签当成生成机制？
有没有把高识别内容形状偷渡进生产？
如果这个结论错了，最可能错在哪里？
这个结论会改变哪一个生产动作？
如果它不改变生产动作，为什么要写进运行包？
```

## A. 小说生产运行包的实际流水线

这一节是实际执行顺序。

不要先问“我要不要写完整世界观”。先按下面这条链跑：

```text
平台机制研究
-> 样本池登记
-> 样本禁区定义
-> 前 10 章证据抽取
-> route_candidate 判断
-> 深读第一卷 / 完整事件链
-> 单书 source study
-> route 横向归纳
-> 内容内核卡
-> runtime route skill 草案
-> clean-room 前 3 章 / 前 10 章计划
-> 盲审 / 追读诊断
-> 叙事身体回修
-> house style
-> 正式项目立项
```

### A.1 平台机制研究

目的：

```text
先知道平台怎么给流量、怎么给钱、怎么签约、怎么处罚，再决定我们要生产什么文本。
```

要研究：

```text
投稿路径：直发、内投、签约、上架、入 V。
收益来源：订阅、广告分成、全勤、补贴、保底、打赏、版权。
触发条件：字数、更新、追读、读完率、均订、收入门槛。
推荐机制：推荐节点、首秀、PK、榜单、编辑判断。
违规损失：断更、抄袭、洗稿、AI 低质、高频发文、合同限制。
新人可达性：新人实际能不能拿到这些钱和流量。
```

产出：

```text
平台机制文档
收益机制清单
投稿路径对比
待验证清单
来源索引与证据等级
```

合格标准：

```text
每条平台结论都有来源等级。
官方规则、作者反馈、媒体线索分开写。
不能把作者评论直接当平台规则。
能反推出 P2 拆书时要记录哪些平台字段。
```

### A.2 样本池入库

目的：

```text
把要研究的书变成可追踪样本，而不是凭记忆聊爆款。
```

每本样本记录：

```yaml
book_id:
title:
author:
platform:
source_url:
sampled_at:
status:
word_count:
category:
tags:
ranking_evidence:
intro_promise:
why_selected:
possible_bias:
access_scope:
```

样本分层：

```text
L0 候选样本：20-30 本，只登记元数据、简介、标签、排名来源。
L1 快拆样本：10-15 本，拆前 10 章。
L2 深拆样本：3-5 本，读前 30 章或完整事件卷。
L3 核心对照：1-2 本，用于 clean-room 诊断和盲审。
```

合格标准：

```text
样本来源清楚。
采样日期清楚。
不是只选我们喜欢的书。
包含成功样本、普通样本和失败/争议样本。
```

### A.3 样本范围与禁止区定义

目的：

```text
在提取前先定义“研究什么”和“什么不能被抽走”。
```

必须先写：

```text
sample_scope
do_not_extract
do_not_transfer
hard_risk
soft_risk
```

典型 do_not_extract：

```text
主角身份组合
专属能力名称
组织名称
怪物 / 异常专属规则
关键桥段顺序
名场面设计
高识别文风
章节标题格式
作者招牌口癖
```

典型 do_not_transfer：

```text
单本书的人设组合
单本书的能力限制
单本书的异常视觉
单本书的反转路径
单本书的组织体系
读者一眼能联想到原书的剧情形状
```

合格标准：

```text
禁止区写在提取之前。
每条证据都能标记迁移边界。
任何高识别内容形状都不能进入 production。
```

### A.4 前 10 章证据快拆

目的：

```text
用最低成本判断一本书前期靠什么留人。
```

前 10 章每章提取：

```yaml
chapter_no:
opening_state:
reader_promise:
protagonist_goal:
trigger_event:
obstacle:
protagonist_action:
information_reveal:
payoff:
state_change:
ending_hook:
next_chapter_pressure:
platform_signal:
transfer_boundary:
risk_note:
```

重点不是写读后感，而是回答：

```text
这一章为什么必须发生？
上一章的结果如何逼出这一章？
主角如果不行动会损失什么？
这章兑现了什么，又留下什么？
章末为什么让读者点下一章？
```

产出：

```text
前 10 章证据表
章节运动表
承诺-兑现表
阻力-反咬表
翻牌/揭示表
```

合格标准：

```text
证据能定位到章节。
每章都有状态变化。
能看出前 10 章小闭环。
不能只写“节奏快”“爽点强”。
```

### A.5 路线候选判断

目的：

```text
前 10 章快拆后，先判断这本书属于哪条生成路线，值不值得深拆。
```

route_candidate 要回答：

```text
这本书的主承诺是什么？
主角主要靠什么赢？
阻力主要从哪里来？
爽点 / 悬疑 / 智斗 / 成长谁是主引擎？
能力体系如何制造剧情？
读者追的是变强、破案、翻牌、关系、身份，还是生存压力？
这条路线新人能不能执行？
这条路线能不能支撑长篇？
相似风险高不高？
```

常见路线候选：

```text
系统升级爽文
异常处置路线
规则破解路线
民俗禁区路线
官方组织任务路线
破绽推演路线
低武高智路线
都市高武资源争夺路线
```

产出：

```text
route_candidate.md
是否进入 L2 深拆决策
放弃 / 观察 / 深拆 标记
```

合格标准：

```text
不深拆不适合我们的书。
不把题材标签当 route。
每个 route_candidate 至少有前 10 章证据支撑。
```

### A.6 Source-open 多颗粒度深读

目的：

```text
补上表格抓不到的整段小说运动。
```

深读颗粒度：

```text
前 3 章：开局承诺和即时留人。
前 10 章：第一个小闭环。
前 30 章：第一轮推荐/追读压力下的中期承接。
完整事件卷：一个事件如何开始、升级、翻牌、收束。
上架前后：付费预期如何转移。
```

必须记录：

```text
开头如何入局
主角如何被迫行动
能力/设定何时出现
第一个真实阻力何时反咬
读者什么时候知道这本书“会怎么爽”
哪些信息被故意藏住
哪些爽点被延迟兑现
事件卷如何收束又打开下一卷
```

产出：

```text
单书 source-open memo
事件链备忘
第一卷运动图
样本失败/强项记录
```

合格标准：

```text
能解释这本书为什么跑得动。
能解释哪些章节删掉会塌。
能解释哪些地方不是优点但是真实追读结构的一部分。
```

### A.7 单书生成模型

目的：

```text
把一本书的证据归纳成“它怎么生成”，不是“它讲了什么”。
```

单书生成模型要写：

```yaml
book_identity:
sample_scope:
reader_promise:
dominant_law:
protagonist_engine:
obstacle_system:
power_system_function:
reveal_system:
payoff_system:
chapter_motion:
volume_motion:
style_constraints:
transferable_mechanisms:
do_not_transfer:
failure_modes:
```

中文说明：

```text
reader_promise：这本书承诺读者追什么。
dominant_law：这本书最优先保护什么效果。
protagonist_engine：主角为什么行动，靠什么持续行动。
obstacle_system：对手、规则、组织、资源、误判如何反咬。
power_system_function：能力体系怎么制造剧情，不只是变强。
reveal_system：什么时候藏信息，什么时候翻牌。
payoff_system：爽点、悬疑、智斗、成长如何兑现。
chapter_motion：章节怎么开、怎么推、怎么钩。
volume_motion：一卷怎么开始、升级、收束、转移。
```

合格标准：

```text
不是剧情梗概。
每条规则能回到原文章节证据。
明确哪些只能观察，不能迁移。
```

### A.8 Route 横向归纳

目的：

```text
从多本书里归纳一条可迁移的打法，而不是复制一本书。
```

进入 route 归纳的前提：

```text
至少 3 本同路线样本。
或者样本不足时明确标为假设，不进入 runtime。
```

route 归纳要回答：

```yaml
route_name:
platform_fit:
primary_reader_promise:
dominant_law:
opening_pattern:
chapter_motion_pattern:
protagonist_win_logic:
obstacle_pattern:
power_system_pattern:
reveal_pattern:
payoff_pattern:
volume_pattern:
required_materials:
newcomer_difficulty:
adaptation_potential:
hard_risks:
soft_risks:
counterexamples:
quality_gate:
```

合格标准：

```text
共性只是候选，不是结论。
必须有差异对照和反例。
不能只总结成功样本，要知道失败样本哪里断。
归纳出来的是机制，不是桥段顺序。
```

### A.9 内容内核卡

目的：

```text
从 route 中抽出真正可迁移的小说成立机制。
```

内容内核卡要回答：

```yaml
kernel_name:
what_reader_chases:
why_it_works:
minimum_required_parts:
causal_loop:
payoff_loop:
obstacle_loop:
reveal_loop:
platform_fit:
transferable_principles:
forbidden_content_shapes:
misuse_risks:
known_failure_modes:
test_prompt:
quality_gate:
```

示例：

```text
异常处置内核
读者追：异常规则、主角破局、危险升级。
最小部件：异常现象、错误解释、观察线索、规则测试、代价、破绽、反制、更大异常。
因果循环：异常制造压力 -> 主角测试 -> 代价暴露规则 -> 主角反制 -> 规则背后还有组织/更大异常。
禁用形状：不能搬某本书的异常视觉、怪物规则、官方组织、任务流程和翻牌顺序。
```

合格标准：

```text
不用样本书名也能描述清楚。
能迁移到新题材、新主角、新设定。
不包含高识别内容资产。
能指导 clean-room 生成。
```

### A.10 Runtime Route Skill 草案

目的：

```text
把 route 和内容内核转成可执行生成协议。
```

skill 草案必须包含：

```text
Activation：什么项目适合启用。
Dominant Law：最优先保护什么阅读效果。
Reader Promise：主承诺和副承诺。
Protagonist Engine：主角行动发动机。
Obstacle System：阻力如何主动反咬。
Power / Setting Function：设定如何制造剧情。
Reveal System：信息怎么藏、怎么放、怎么翻牌。
Chapter Engine：每章怎么开、怎么推、怎么收。
Volume Engine：第一卷怎么闭合和转移。
Style Boundary：最低文风约束。
Transfer Boundary：不能迁移什么。
Drift Alarms：什么时候已经偏成水文或换皮。
Validation Gate：怎么判断能不能用。
```

合格标准：

```text
每条规则有证据来源。
明确 diagnostic / candidate / stable 状态。
不能把单书特色写成 route 规则。
```

### A.11 Clean-room 开局诊断

目的：

```text
用隔离生成测试 route/内核是否真的能跑。
```

写手允许读取：

```text
内容内核卡
runtime route skill 草案
平台约束
合规边界
我们自己的初始项目设想
```

写手禁止读取：

```text
原书全文
高识别桥段
具体章节顺序
样本书角色/能力/组织/异常规则
```

产出：

```text
前 3 章计划
前 10 章计划
第一事件卷草案
风险自查
```

合格标准：

```text
不是某本书换皮。
主角行动有因果。
阻力会反咬。
前 3 章有明确兑现。
前 10 章有小闭环。
读者知道为什么要追下一章。
```

注意：

```text
clean-room 开局是诊断，不是最终正文。
它暴露的问题要回到样本证据和内核卡修，不是靠加 prompt 硬补。
```

### A.12 盲审 / 追读诊断

目的：

```text
用外部视角判断生成稿是否有追读、是否自嗨、是否像某本书。
```

盲审问题：

```text
你愿不愿意看第 4 章？为什么？
你觉得主角为什么必须行动？
你觉得最大阻力是什么？
哪一处最像水文？
哪一处最像某本书或某类套路？
你是否看懂能力/异常规则？
章末钩子有没有真实压力？
如果弃书，你会在哪一章弃？
```

诊断分类：

```text
route 错：主承诺不成立。
内核弱：机制没有带动剧情。
因果断：下一章不是被上一章逼出来的。
阻力假：敌人/规则不会主动反咬。
翻牌假：信息释放没有改变局势。
文风弱：读起来像 AI 水文。
相似风险：像某本书贴皮。
平台不适配：节奏或承诺不符合当前平台。
```

合格标准：

```text
盲审反馈能定位到章节和原因。
不能只收“好看/不好看”。
盲审反馈不能直接改 skill，必须回到证据确认。
```

### A.13 叙事身体抽取与回修

目的：

```text
如果 clean-room 失败，不是继续堆提示词，而是抽“小说真实怎么动”。
```

要抽：

```text
章节进入方式
主角被迫行动的位置
阻力第一次反咬的位置
线索出现密度
误判和纠正节奏
爽点兑现位置
翻牌改变局势的位置
章末钩子类型
段落和场景长度
对白 / 动作 / 心理比例
```

对失败稿要看：

```text
哪里像大纲，不像小说。
哪里像设定说明，不像剧情。
哪里主角只是被作者推着走。
哪里阻力只是摆设。
哪里翻牌只是解释，不改变局势。
哪里文风像 AI。
```

产出：

```text
叙事身体差异表
失败层级记录
route skill 修订项
Style Bible 修订项
```

合格标准：

```text
能回答为什么失败。
能区分是 route 问题、内核问题、章节因果问题、文风问题还是相似风险。
下一轮改动来自证据，不来自感觉。
```

### A.14 House Style / Style Bible

目的：

```text
形成我们自己的小说声音，而不是在“不能像别人”和“AI 默认文风”之间摇摆。
```

House Style 要回答：

```text
我们叙述距离多近？
句子偏短促还是偏细密？
动作、对白、心理、环境的比例是什么？
爽点能写多重？
智斗解释能写多细？
民俗/高武/异常的真实感边界在哪里？
什么算太中二？
什么算太学院？
什么算太水？
什么算太像某本书？
```

当前都市高武/异常事件可先设为：

```text
冷静、压迫、规则感强。
动作清楚，但不无限堆战力。
异常有可观察规则，不靠玄学乱变。
主角胜利来自观察、试探、能力代价和破绽利用。
每次爽点都要改变处境。
```

产出：

```text
Style Bible
对白规则
场景规则
动作规则
心理描写规则
异常描写规则
禁用文风清单
```

合格标准：

```text
生成稿不像单一对标书。
读者能感到稳定项目声音。
文风服务追读，不单纯追求漂亮。
```

### A.15 独立生产与评估回写

目的：

```text
用我们自己的项目设定、内容内核和 house style 写正式稿。
```

写稿前必须有：

```yaml
project_positioning:
platform:
route:
content_kernel:
reader_promise:
protagonist_engine:
obstacle_system:
power_system:
first_event_chain:
house_style:
forbidden_similarity_shapes:
quality_gate:
```

评估要看：

```text
能不能追读。
有没有前 3 章即时兑现。
有没有前 10 章小闭环。
主角行动是否被逼出来。
阻力是否会反咬。
翻牌是否改变局势。
能力是否制造剧情。
文风是否稳定。
是否像某本书。
是否符合平台收益/推荐机制。
```

产出：

```text
正式前 3 章
正式前 10 章
第一卷事件链
评估报告
回修记录
```

合格标准：

```text
不是复刻稿。
不是泛泛 AI 水文。
不是单本书换皮。
能作为我们自己的小说项目进入平台验证。
```

## 1. 为什么小说必须这样改

短视频文案的核心目标是：

```text
讲清一个问题，并让观众听完。
```

小说的核心目标是：

```text
让读者持续点下一章，并相信这本书能长期兑现。
```

所以小说不能只抽：

```text
结构
文风
爽点
设定
```

还必须抽：

```text
承诺如何建立
因果如何逼出下一幕
阻力如何反咬
信息如何揭示
能力如何制造剧情
章节如何续读
第一卷如何闭合
```

如果缺这一层，就会出现：

```text
设定很好，但主角不动。
文风顺了，但剧情不推。
开头有爽点，但第 4 章不知道写什么。
像某本爆款，但有换皮风险。
不像任何书，但像低质量 AI 水文。
```

## 2. 核心概念

### 2.1 对标书

对标书只回答：

```text
这种路线怎么跑？
读者为什么追？
平台为什么给量？
哪些机制可以迁移？
哪些内容不能碰？
```

它不回答：

```text
我们要不要写同样的主角。
我们要不要写同样的能力。
我们要不要按它的章节顺序来。
我们要不要模仿作者文风。
```

### 2.2 Route / 路线模型

route 不是题材。

```text
都市高武是题材。
异常处置、规则破解、系统升级、破绽推演，才是 route。
```

route 决定：

```text
读者追什么。
主角怎么赢。
阻力怎么反咬。
爽点怎么兑现。
信息怎么翻牌。
能力怎么持续制造剧情。
```

### 2.3 内容内核

内容内核是 route 背后的最小可迁移生成机制。

它要能脱离样本书名成立。

错误写法：

```text
学习《某书》的官方组织 + 异常任务 + 主角能力。
```

正确写法：

```text
学习“异常处置路线”中，异常规则如何先制造误判，再通过测试代价暴露破绽，最后让主角用有限能力反制。
```

### 2.4 叙事身体

叙事身体是小说真实移动的方式。

它不是大纲，也不是文风。

它包括：

```text
场景怎么进
信息怎么流
人物怎么动
阻力怎么压
爽点怎么落
钩子怎么抛
读者怎么被推到下一章
```

很多 AI 文最大的问题不是设定不够，而是没有叙事身体。

### 2.5 House Style

house style 是我们自己的小说声音。

它不是“不要像别人”这么简单。

它要规定：

```text
我们怎么写动作
怎么写对白
怎么写心理
怎么写异常
怎么写能力
怎么写爽点
怎么写章末
怎么避免 AI 味
怎么避免贴皮感
```

## 3. 晋级规则

### 3.1 样本进入 L1 的条件

```text
来源清楚。
平台和标签清楚。
简介承诺明确。
不是明显无关样本。
相似风险可控。
```

### 3.2 样本进入 L2 的条件

```text
前 10 章能看出明确路线。
有可学习机制。
不是单靠独特设定或作者粉丝跑起来。
相似风险没有过高。
```

### 3.3 Route 进入候选的条件

```text
至少 2-3 本样本出现类似机制。
能解释成功样本。
能解释失败样本。
能被 clean-room 写出新开局。
```

### 3.4 内容内核进入候选的条件

```text
不用书名也能讲清。
不含高识别内容形状。
能迁移到我们的项目设想。
有明确误用风险和失败模式。
```

### 3.5 Runtime skill 进入可用的条件

```text
每条规则有证据。
通过 clean-room 开局诊断。
通过相似风险检查。
能产出前 10 章计划。
```

### 3.6 正式生产允许条件

```text
有项目定位书。
有内容内核卡。
有 Story Bible。
有 Style Bible / house style。
有合规和相似风险边界。
前 3 章 / 前 10 章通过盲审或内部追读诊断。
```

## 4. 当前项目的最小闭环

当前不要全量搭系统，先跑最小闭环：

```text
P0.6 三平台机制研究
-> P1 L0 样本池
-> P2 选 3 本做前 10 章快拆
-> P2.5 判断 1-2 条 route_candidate
-> P3 对 1 条 route 做 3 本深读
-> P4 归纳 1 张内容内核卡
-> P5 写 1 版 runtime route skill 草案
-> P6 clean-room 生成前 3 章计划
-> P7 盲审/追读诊断
-> 决定继续、改 route，还是放弃
```

最小闭环只验证一个问题：

```text
我们能不能从多本样本里抽出一个不换皮、能追读、可生产的小说生成机制。
```

如果这个闭环跑不通，不要继续扩写世界观。

## 5. 工作原则

```text
样本是显微镜。
提取是找证据。
归纳是找机制。
clean-room 是诊断。
盲审是暴露缺口。
内容内核是迁移对象。
house style 是我们的声音。
正式生产必须脱离单一样本。
```

更短一句：

```text
不借鉴一本书的形状。
不凭空写 AI 水文。
只融合抽象生成机制。
```

## 6. Critical-thinking 自问自答审查

这一节是对本文档本身的审查。它不是最终结论，而是后续回修清单。

### Q1：这个文档是不是已经可以直接执行？

答：

```text
可以作为路线图执行，但还不能算完整操作系统。
```

原因：

```text
它已经写清了阶段、目的、产出和合格标准。
但还缺少可直接填写的模板文件、字段枚举、样本记录表和盲审记录表。
```

下一步：

```text
补 4 个模板：
1. sample_record.yaml
2. chapter_evidence.yaml
3. content_kernel_card.md
4. blind_review_report.md
```

### Q2：它最大的漏洞是什么？

答：

```text
最大漏洞是：流程完整，但执行成本可能过高。
```

风险：

```text
如果每本书都按全流程跑，会在正式写作前耗死。
```

修正：

```text
必须坚持最小闭环。
先用 3 本样本跑一条 route，不要一上来做完整路线库。
```

### Q3：它有没有把“番茄当前机制”误当小说通用能力？

答：

```text
目前有防线，但还不够硬。
```

已有防线：

```text
文档明确番茄只是当前平台约束。
```

缺口：

```text
后续样本如果只采番茄，route 很容易被番茄推荐机制塑形。
```

修正：

```text
番茄样本用于第一训练场。
七猫用于编辑商业判断对照。
起点用于付费追读和长线可信度对照。
```

### Q4：它有没有解决“借鉴很多本”和“避免洗稿”的矛盾？

答：

```text
方向解决了，但还需要模板强制执行。
```

正确边界：

```text
借鉴抽象生成机制。
不借鉴高识别内容形状。
```

仍需补强：

```text
每条 evidence 都必须有 transfer_boundary。
每张内容内核卡都必须有 forbidden_content_shapes。
每次 clean-room 后必须做相似风险检查。
```

### Q5：它有没有足够重视文笔/文风？

答：

```text
有写 house style，但还不够操作化。
```

缺口：

```text
当前 house style 还停在原则层。
还没拆成句法、段落、对白、动作、心理、环境、章末气口等可执行字段。
```

修正：

```text
后续需要单独补 Style Bible 模板。
文风不能最后随便补，也不能早于 route 和内容内核抽取。
正确位置是在 clean-room 诊断后、正式正文前。
```

### Q6：它有没有把 clean-room 误写成“复刻”？

答：

```text
已经明确 clean-room 是诊断，不是生产。
```

但仍要注意：

```text
小说侧不要做“像不像某本书”的盲审。
应该做“是否追读、是否像换皮、是否有 AI 水文感、是否能解释下一章压力”的盲审。
```

### Q7：它有没有缺少“什么情况下停止”的判断？

答：

```text
目前有晋级规则，但停止规则不够明确。
```

需要补：

```text
如果前 10 章快拆无法提取明确 route，停止深拆。
如果 route 只能由一本书支撑，停止升级 runtime。
如果 clean-room 连续两轮无法产生追读，停止扩写世界观。
如果相似风险无法降到可控，停止该 route。
```

### Q8：它现在最应该补什么，而不是继续扩理论？

答：

```text
最应该补模板和执行门，不是继续写方法论。
```

优先级：

```text
1. 补 4 个模板。
2. 给 P2 前10章快拆做 3 本校准批次。
3. 用校准结果回修字段。
4. 再决定是否进入 P3 深拆。
```
