# 小说内容内核发现系统操作方案

## 0. 一句话模型

```text
对标书不是目标，而是显微镜。
clean-room 开局不是生产，而是诊断工具。
真正目标是：从多本样本中发掘可迁移的小说内容内核，把内核适配到目标平台和题材，再用我们自己的 Story Bible / Style Bible 生产。
```

完整流程：

```text
平台 / 赛道研究
-> 样本池入库
-> 合规与相似风险边界
-> 前 10 章证据快拆
-> 路线候选判断
-> source-open 多颗粒度深读
-> 单书生成模型
-> route 横向归纳
-> 受控 clean-room 开局诊断
-> 盲审 / 追读诊断
-> 叙事身体抽取
-> 内容内核卡
-> 平台 / 赛道适配
-> 自有 Story Bible / Style Bible
-> 独立生产
-> 评估与回写
```

这套系统要同时保留两件事：

```text
旧模式的锋利反馈：clean-room + 盲审作为诊断显微镜。
新模式的安全边界：不把某本书、某作者、某个爆款开局当生产目标。
```

### 0.1.3 critical-thinking-mode 的人工执行规则

在正式 skill 补齐前，每个关键节点必须人工执行以下自问自答：

```text
我现在的结论依赖什么证据？
有没有反例？
有没有把单本书特色当成 route 规律？
有没有把题材标签当成生成机制？

这个结论会改变哪一个生产动作？
如果它不改变生产动作，为什么要写进运行包？
```

## A. 小说生产运行包的实际流水线

这一节是实际执行顺序。

不要先问“我要不要写完整世界观”。先按下面这条链跑：

```text
平台机制研究
-> 样本池登记
-> 样本禁区定义
-> 前 10 章证据抽取
-> route_candidate 判断
-> 深读第一卷 / 完整事件链
-> 单书 source study
-> route 横向归纳
-> 内容内核卡
-> runtime route skill 草案
-> clean-room 前 3 章 / 前 10 章计划
-> 盲审 / 追读诊断
-> 叙事身体回修
-> house style
-> 正式项目立项
```

## B. 小说按什么维度归纳和诊断

短视频文案通常可以按“单条视频长度”归纳：一条视频就是一个相对完整的内容单元。

小说不行。小说太长，不能只按“整本书”归纳，也不能只按“章节”归纳。小说必须按多颗粒度矩阵归纳。

### B.1 归纳颗粒度

| 颗粒度 | 看什么 | 作用 | 不能做什么 |
| --- | --- | --- | --- |
| 单章 | 开局处境、触发、行动、阻力、兑现、钩子、状态变化 | 判断这一章为什么让读者继续点下一章 | 不能用单章判断整本书能力 |
| 前 3 章 | 入口承诺、主角亮相、即时爽点/悬疑、核心卖点 | 判断开局能不能留住第一批读者 | 不能证明长篇可持续 |
| 前 10 章 | 第一个小闭环、主角发动机、阻力反咬、初次翻牌 | 判断 route_candidate 是否成立 | 不能直接升级为正式 route |
| 前 30 章 | 推荐/追读压力下的承接、能力展开、关系和组织进入 | 判断中期是否开始掉速或水化 | 不能代替完整第一卷 |
| 完整事件链/副本 | 事件如何开始、升级、误导、翻牌、收束、转入下一事件 | 判断叙事因果和翻牌系统是否真实工作 | 不能只抽设定，不看事件闭合 |
| 第一卷 | 主承诺如何建立、兑现、升级和转移 | 判断长篇开局是否有可持续主线 | 不能照搬卷结构 |
| 上架前后 | 免费追读如何转成付费/长期追读 | 用于起点等付费生态对照 | 不能用番茄逻辑直接解释 |
| 整本书/长线 | 主线、升级、人物变化、世界展开、完本兑现 | 判断长期生产能力和 IP 潜力 | 不适合作为第一轮低成本拆解 |

### B.2 归纳维度

每个颗粒度都不是只看结构，而是至少看 9 个维度：

| 维度 | 核心问题 | 主要颗粒度 |
| --- | --- | --- |
| 读者承诺 | 读者追的到底是什么，主承诺和副承诺是什么 | 简介、前3章、前10章、第一卷 |
| 章节结构 | 每章如何进入、推进、兑现、留钩子 | 单章、前10章 |
| 叙事因果 | 上一幕如何逼出下一幕，主角为什么不能不行动 | 单章、事件链、第一卷 |
| 主角发动机 | 主角欲望、短板、能力、风险如何持续制造行动 | 前3章、前10章、第一卷 |
| 阻力系统 | 敌人、规则、组织、资源、误判、关系如何反咬 | 单章、事件链、第一卷 |
| 揭示/翻牌 | 信息什么时候藏、什么时候放、什么时候改写局势 | 前10章、事件链、第一卷 |
| 能力/设定功能 | 能力和设定如何制造限制、线索、代价、反制和新问题 | 前10章、前30章、事件链 |
| 叙事身体 | 场景怎么进、信息怎么流、人物怎么动、章末怎么推 | 单章、事件链 |
| 文风/渲染器 | 叙述距离、句段节奏、对白、动作、心理、环境比例 | 单章、前3章、clean-room 后 |

### B.3 诊断/复刻颗粒度

小说不做“整本复刻”。小说的 clean-room 诊断分层做：

| 诊断层级 | 生成什么 | 诊断什么 | 通过标准 |
| --- | --- | --- | --- |
| 单章诊断 | 写 1 章或 1 个场景 | 叙事身体、场景推进、文风是否像 AI 水文 | 有目标、阻力、动作、状态变化 |
| 前 3 章诊断 | 写前 3 章计划或样章 | 入口承诺、即时兑现、主角发动机 | 读者知道为什么要看第 4 章 |
| 前 10 章诊断 | 写前 10 章计划 | 小闭环、阻力反咬、翻牌、追读压力 | 有明确 route 运动，不是设定堆叠 |
| 事件链诊断 | 写一个完整事件/副本大纲 | 因果链、误导、反制、收束 | 事件结束后局势改变，并推出下一事件 |
| 第一卷诊断 | 写第一卷结构 | 主承诺兑现和升级 | 能解释为什么能继续写 30-50 万字 |
| Style 诊断 | 写同一场景的不同渲染版本 | house style 是否稳定 | 不像单一对标书，也不像 AI 默认文 |

诊断不是为了“像样本”，而是为了定位问题：

```text
前 3 章失败，多半是承诺/主角发动机问题。
前 10 章失败，多半是 route/阻力/小闭环问题。
事件链失败，多半是因果/翻牌/收束问题。
样章失败，多半是叙事身体/文风/AI 味问题。
第一卷失败，多半是长期承诺和生产能力问题。
```

### B.4 当前阶段采用的最小颗粒度

当前项目不直接做整本书归纳，也不先做全文复刻。

当前采用：

```text
L0：样本元数据和简介承诺
L1：前 10 章快拆
L2：前 30 章或完整事件链深读
L3：clean-room 前 3 章 / 前 10 章诊断
```

原因：

```text
前 10 章足够判断 route_candidate。
完整事件链足够判断叙事因果和翻牌系统。
前 3 章 / 前 10 章 clean-room 足够暴露我们能不能写出追读。
整本书长线归纳放到 route 通过后再做。
```

如果这几个颗粒度都跑不通，不进入整本书规划。

### A.1 平台机制研究

目的：

```text
先知道平台怎么给流量、怎么给钱、怎么签约、怎么处罚，再决定我们要生产什么文本。
```

要研究：

```text
投稿路径：直发、内投、签约、上架、入 V。
收益来源：订阅、广告分成、全勤、补贴、保底、打赏、版权。
触发条件：字数、更新、追读、读完率、均订、收入门槛。
推荐机制：推荐节点、首秀、PK、榜单、编辑判断。
违规损失：断更、抄袭、洗稿、AI 低质、高频发文、合同限制。
新人可达性：新人实际能不能拿到这些钱和流量。
```

产出：

```text
平台机制文档
收益机制清单
投稿路径对比
待验证清单
来源索引与证据等级
```

合格标准：

```text
每条平台结论都有来源等级。
官方规则、作者反馈、媒体线索分开写。
不能把作者评论直接当平台规则。
能反推出 P2 拆书时要记录哪些平台字段。
```

### A.2 样本池入库

目的：

```text
把要研究的书变成可追踪样本，而不是凭记忆聊爆款。
```

每本样本记录：

```yaml
book_id:
title:
author:
platform:
source_url:
sampled_at:
status:
word_count:
category:
tags:
ranking_evidence:
intro_promise:
why_selected:
possible_bias:
access_scope:
```

样本分层：

```text
L0 候选样本：20-30 本，只登记元数据、简介、标签、排名来源。
L1 快拆样本：10-15 本，拆前 10 章。
L2 深拆样本：3-5 本，读前 30 章或完整事件卷。
L3 核心对照：1-2 本，用于 clean-room 诊断和盲审。
```

合格标准：

```text
样本来源清楚。
采样日期清楚。
不是只选我们喜欢的书。
包含成功样本、普通样本和失败/争议样本。
```

### A.3 样本范围与禁止区定义

目的：

```text
在提取前先定义“研究什么”和“什么不能被抽走”。
```

必须先写：

```text
sample_scope
do_not_extract
do_not_transfer
hard_risk
soft_risk
```

典型 do_not_extract：

```text
主角身份组合
专属能力名称
组织名称
怪物 / 异常专属规则
关键桥段顺序
名场面设计
高识别文风
章节标题格式
作者招牌口癖
```

典型 do_not_transfer：

```text
单本书的人设组合
单本书的能力限制
单本书的异常视觉
单本书的反转路径
单本书的组织体系
读者一眼能联想到原书的剧情形状
```

合格标准：

```text
禁止区写在提取之前。
每条证据都能标记迁移边界。
任何高识别内容形状都不能进入 production。
```

### A.4 前 10 章证据快拆

目的：

```text
用最低成本判断一本书前期靠什么留人。
```

前 10 章每章提取：

```yaml
chapter_no:
opening_state:
reader_promise:
protagonist_goal:
trigger_event:
obstacle:
protagonist_action:
information_reveal:
payoff:
state_change:
ending_hook:
next_chapter_pressure:
platform_signal:
transfer_boundary:
risk_note:
```

重点不是写读后感，而是回答：

```text
这一章为什么必须发生？
上一章的结果如何逼出这一章？
主角如果不行动会损失什么？
这章兑现了什么，又留下什么？
章末为什么让读者点下一章？
```

产出：

```text
前 10 章证据表
章节运动表
承诺-兑现表
阻力-反咬表
翻牌/揭示表
```

合格标准：

```text
证据能定位到章节。
每章都有状态变化。
能看出前 10 章小闭环。
不能只写“节奏快”“爽点强”。
```

### A.5 路线候选判断

目的：

```text
前 10 章快拆后，先判断这本书属于哪条生成路线，值不值得深拆。
```

route_candidate 要回答：

```text
这本书的主承诺是什么？
主角主要靠什么赢？
阻力主要从哪里来？
爽点 / 悬疑 / 智斗 / 成长谁是主引擎？
能力体系如何制造剧情？
读者追的是变强、破案、翻牌、关系、身份，还是生存压力？
这条路线新人能不能执行？
这条路线能不能支撑长篇？
相似风险高不高？
```

常见路线候选：

```text
系统升级爽文
异常处置路线
规则破解路线
民俗禁区路线
官方组织任务路线
破绽推演路线
低武高智路线
都市高武资源争夺路线
```

产出：

```text
route_candidate.md
是否进入 L2 深拆决策
放弃 / 观察 / 深拆 标记
```

合格标准：

```text
不深拆不适合我们的书。
不把题材标签当 route。
每个 route_candidate 至少有前 10 章证据支撑。
```

### A.6 Source-open 多颗粒度深读

目的：

```text
补上表格抓不到的整段小说运动。
```

深读颗粒度：

```text
前 3 章：开局承诺和即时留人。
前 10 章：第一个小闭环。
前 30 章：第一轮推荐/追读压力下的中期承接。
完整事件卷：一个事件如何开始、升级、翻牌、收束。
上架前后：付费预期如何转移。
```

必须记录：

```text
开头如何入局
主角如何被迫行动
能力/设定何时出现
第一个真实阻力何时反咬
读者什么时候知道这本书“会怎么爽”
哪些信息被故意藏住
哪些爽点被延迟兑现
事件卷如何收束又打开下一卷
```

产出：

```text
单书 source-open memo
事件链备忘
第一卷运动图
样本失败/强项记录
```

合格标准：

```text
能解释这本书为什么跑得动。
能解释哪些章节删掉会塌。
能解释哪些地方不是优点但是真实追读结构的一部分。
```

单书生成模型要写：

```yaml

power_system_function:
reveal_system:
payoff_system:

transferable_mechanisms:
do_not_transfer:
failure_modes:
```

中文说明：

```text
reader_promise：这本书承诺读者追什么。
dominant_law：这本书最优先保护什么效果。
protagonist_engine：主角为什么行动，靠什么持续行动。
obstacle_system：对手、规则、组织、资源、误判如何反咬。
power_system_function：能力体系怎么制造剧情，不只是变强。
reveal_system：什么时候藏信息，什么时候翻牌。
payoff_system：爽点、悬疑、智斗、成长如何兑现。
chapter_motion：章节怎么开、怎么推、怎么钩。
volume_motion：一卷怎么开始、升级、收束、转移。
```

合格标准：

```text
不是剧情梗概。
每条规则能回到原文章节证据。
明确哪些只能观察，不能迁移。
```

### A.8 Route 横向归纳

目的：

```text
从多本书里归纳一条可迁移的打法，而不是复制一本书。
```

进入 route 归纳的前提：

```text
至少 3 本同路线样本。
或者样本不足时明确标为假设，不进入 runtime。
```

route 归纳要回答：

```yaml
route_name:
platform_fit:
primary_reader_promise:
dominant_law:
opening_pattern:
chapter_motion_pattern:
protagonist_win_logic:
obstacle_pattern:
power_system_pattern:
reveal_pattern:
payoff_pattern:
volume_pattern:
required_materials:
newcomer_difficulty:
adaptation_potential:
hard_risks:
soft_risks:
counterexamples:
quality_gate:
```

合格标准：

```text
共性只是候选，不是结论。
必须有差异对照和反例。
不能只总结成功样本，要知道失败样本哪里断。
归纳出来的是机制，不是桥段顺序。
```

### A.9 内容内核卡

目的：

```text
从 route 中抽出真正可迁移的小说成立机制。
```

内容内核卡要回答：

```yaml
kernel_name:
what_reader_chases:
why_it_works:
minimum_required_parts:
causal_loop:
payoff_loop:
obstacle_loop:
reveal_loop:
platform_fit:
transferable_principles:
forbidden_content_shapes:
misuse_risks:
known_failure_modes:
test_prompt:
quality_gate:
```

示例：

```text
异常处置内核
读者追：异常规则、主角破局、危险升级。
最小部件：异常现象、错误解释、观察线索、规则测试、代价、破绽、反制、更大异常。
因果循环：异常制造压力 -> 主角测试 -> 代价暴露规则 -> 主角反制 -> 规则背后还有组织/更大异常。
禁用形状：不能搬某本书的异常视觉、怪物规则、官方组织、任务流程和翻牌顺序。
```

合格标准：

```text
不用样本书名也能描述清楚。
能迁移到新题材、新主角、新设定。
不包含高识别内容资产。
能指导 clean-room 生成。
```

### A.10 Runtime Route Skill 草案

目的：

```text
把 route 和内容内核转成可执行生成协议。
```

skill 草案必须包含：

```text
Activation：什么项目适合启用。
Dominant Law：最优先保护什么阅读效果。
Reader Promise：主承诺和副承诺。
Protagonist Engine：主角行动发动机。
Obstacle System：阻力如何主动反咬。
Power / Setting Function：设定如何制造剧情。
Reveal System：信息怎么藏、怎么放、怎么翻牌。
Chapter Engine：每章怎么开、怎么推、怎么收。
Volume Engine：第一卷怎么闭合和转移。
Style Boundary：最低文风约束。
Transfer Boundary：不能迁移什么。
Drift Alarms：什么时候已经偏成水文或换皮。
Validation Gate：怎么判断能不能用。
```

合格标准：

```text
每条规则有证据来源。
明确 diagnostic / candidate / stable 状态。
不能把单书特色写成 route 规则。
```

### A.11 Clean-room 开局诊断

目的：

```text
用隔离生成测试 route/内核是否真的能跑。
```

写手允许读取：

```text
内容内核卡
runtime route skill 草案
平台约束
合规边界
我们自己的初始项目设想
```

写手禁止读取：

```text
原书全文
高识别桥段
具体章节顺序
样本书角色/能力/组织/异常规则
```

产出：

```text
前 3 章计划
前 10 章计划
第一事件卷草案
风险自查
```

合格标准：

```text
不是某本书换皮。
主角行动有因果。
阻力会反咬。
前 3 章有明确兑现。
前 10 章有小闭环。
读者知道为什么要追下一章。
```

注意：

```text
clean-room 开局是诊断，不是最终正文。
它暴露的问题要回到样本证据和内核卡修，不是靠加 prompt 硬补。
```

### A.12 盲审 / 追读诊断

目的：

```text
用外部视角判断生成稿是否有追读、是否自嗨、是否像某本书。
```

盲审问题：

```text
你愿不愿意看第 4 章？为什么？
你觉得主角为什么必须行动？
你觉得最大阻力是什么？
哪一处最像水文？
哪一处最像某本书或某类套路？
你是否看懂能力/异常规则？
章末钩子有没有真实压力？
如果弃书，你会在哪一章弃？
```

诊断分类：

```text
route 错：主承诺不成立。
内核弱：机制没有带动剧情。
因果断：下一章不是被上一章逼出来的。
阻力假：敌人/规则不会主动反咬。
翻牌假：信息释放没有改变局势。
文风弱：读起来像 AI 水文。
相似风险：像某本书贴皮。
平台不适配：节奏或承诺不符合当前平台。
```

合格标准：

```text
盲审反馈能定位到章节和原因。
不能只收“好看/不好看”。
盲审反馈不能直接改 skill，必须回到证据确认。
```

### A.13 叙事身体抽取与回修

目的：

```text
如果 clean-room 失败，不是继续堆提示词，而是抽“小说真实怎么动”。
```

要抽：

```text
章节进入方式
主角被迫行动的位置
阻力第一次反咬的位置
线索出现密度
误判和纠正节奏
爽点兑现位置
翻牌改变局势的位置
章末钩子类型
段落和场景长度
对白 / 动作 / 心理比例
```

对失败稿要看：

```text
哪里像大纲，不像小说。
哪里像设定说明，不像剧情。
哪里主角只是被作者推着走。
哪里阻力只是摆设。
哪里翻牌只是解释，不改变局势。
哪里文风像 AI。
```

产出：

```text
叙事身体差异表
失败层级记录
route skill 修订项
Style Bible 修订项
```

合格标准：

```text
能回答为什么失败。
能区分是 route 问题、内核问题、章节因果问题、文风问题还是相似风险。
下一轮改动来自证据，不来自感觉。
```

### A.14 House Style / Style Bible

目的：

```text
形成我们自己的小说声音，而不是在“不能像别人”和“AI 默认文风”之间摇摆。
```

House Style 要回答：

```text
我们叙述距离多近？
句子偏短促还是偏细密？
动作、对白、心理、环境的比例是什么？
爽点能写多重？
智斗解释能写多细？
民俗/高武/异常的真实感边界在哪里？
什么算太中二？
什么算太学院？
什么算太水？
什么算太像某本书？
```

当前都市高武/异常事件可先设为：

```text
冷静、压迫、规则感强。
动作清楚，但不无限堆战力。
异常有可观察规则，不靠玄学乱变。
主角胜利来自观察、试探、能力代价和破绽利用。
每次爽点都要改变处境。
```

产出：

```text
Style Bible
对白规则
场景规则
动作规则
心理描写规则
异常描写规则
禁用文风清单
```

合格标准：

```text
生成稿不像单一对标书。
读者能感到稳定项目声音。
文风服务追读，不单纯追求漂亮。
```

### A.15 独立生产与评估回写

目的：

```text
用我们自己的项目设定、内容内核和 house style 写正式稿。
```

写稿前必须有：

```yaml
project_positioning:
platform:
route:
content_kernel:
reader_promise:
protagonist_engine:
obstacle_system:
power_system:
first_event_chain:
house_style:
forbidden_similarity_shapes:
quality_gate:
```

评估要看：

```text
能不能追读。
有没有前 3 章即时兑现。
有没有前 10 章小闭环。
主角行动是否被逼出来。
阻力是否会反咬。
翻牌是否改变局势。
能力是否制造剧情。
文风是否稳定。
是否像某本书。
是否符合平台收益/推荐机制。
```

产出：

```text
正式前 3 章
正式前 10 章
第一卷事件链
评估报告
回修记录
```

合格标准：

```text
不是复刻稿。
不是泛泛 AI 水文。
不是单本书换皮。
能作为我们自己的小说项目进入平台验证。
```

## 1. 为什么小说必须这样改

短视频文案的核心目标是：

```text
讲清一个问题，并让观众听完。
```

小说的核心目标是：

```text
让读者持续点下一章，并相信这本书能长期兑现。
```

所以小说不能只抽：

```text
结构
文风
爽点
设定
```

还必须抽：

```text
承诺如何建立
因果如何逼出下一幕
阻力如何反咬
信息如何揭示
能力如何制造剧情
章节如何续读
第一卷如何闭合
```

如果缺这一层，就会出现：

```text
设定很好，但主角不动。
文风顺了，但剧情不推。
开头有爽点，但第 4 章不知道写什么。
像某本爆款，但有换皮风险。
不像任何书，但像低质量 AI 水文。
```

## 2. 核心概念

### 2.1 对标书

对标书只回答：

```text
这种路线怎么跑？
读者为什么追？
平台为什么给量？
哪些机制可以迁移？
哪些内容不能碰？
```

它不回答：

```text
我们要不要写同样的主角。
我们要不要写同样的能力。
我们要不要按它的章节顺序来。
我们要不要模仿作者文风。
```

### 2.2 Route / 路线模型

route 不是题材。

```text
都市高武是题材。
异常处置、规则破解、系统升级、破绽推演，才是 route。
```

route 决定：

```text
读者追什么。
主角怎么赢。
阻力怎么反咬。
爽点怎么兑现。
信息怎么翻牌。
能力怎么持续制造剧情。
```

### 2.3 内容内核

内容内核是 route 背后的最小可迁移生成机制。

它要能脱离样本书名成立。

错误写法：

```text
学习《某书》的官方组织 + 异常任务 + 主角能力。
```

正确写法：

```text
学习“异常处置路线”中，异常规则如何先制造误判，再通过测试代价暴露破绽，最后让主角用有限能力反制。
```

### 2.4 叙事身体

叙事身体是小说真实移动的方式。

它不是大纲，也不是文风。

它包括：

```text
场景怎么进
信息怎么流
人物怎么动
阻力怎么压
爽点怎么落
钩子怎么抛
读者怎么被推到下一章
```

很多 AI 文最大的问题不是设定不够，而是没有叙事身体。

### 2.5 House Style

house style 是我们自己的小说声音。

它不是“不要像别人”这么简单。

它要规定：

```text
我们怎么写动作
怎么写对白
怎么写心理
怎么写异常
怎么写能力
怎么写爽点
怎么写章末
怎么避免 AI 味
怎么避免贴皮感
```

## 3. 晋级规则

### 3.1 样本进入 L1 的条件

```text
来源清楚。
平台和标签清楚。
简介承诺明确。
不是明显无关样本。
相似风险可控。
```

### 3.2 样本进入 L2 的条件

```text
前 10 章能看出明确路线。
有可学习机制。
不是单靠独特设定或作者粉丝跑起来。
相似风险没有过高。
```

### 3.3 Route 进入候选的条件

```text
至少 2-3 本样本出现类似机制。
能解释成功样本。
能解释失败样本。
能被 clean-room 写出新开局。
```

### 3.4 内容内核进入候选的条件

```text
不用书名也能讲清。
不含高识别内容形状。
能迁移到我们的项目设想。
有明确误用风险和失败模式。
```

### 3.5 Runtime skill 进入可用的条件

```text
每条规则有证据。
通过 clean-room 开局诊断。
通过相似风险检查。
能产出前 10 章计划。
```

### 3.6 正式生产允许条件

```text
有项目定位书。
有内容内核卡。
有 Story Bible。
有 Style Bible / house style。
有合规和相似风险边界。
前 3 章 / 前 10 章通过盲审或内部追读诊断。
```

## 4. 当前项目的最小闭环

当前不要全量搭系统，先跑最小闭环：

```text
P0.6 三平台机制研究
-> P1 L0 样本池
-> P2 选 3 本做前 10 章快拆
-> P2.5 判断 1-2 条 route_candidate
-> P3 对 1 条 route 做 3 本深读
-> P4 归纳 1 张内容内核卡
-> P5 写 1 版 runtime route skill 草案
-> P6 clean-room 生成前 3 章计划
-> P7 盲审/追读诊断
-> 决定继续、改 route，还是放弃
```

最小闭环只验证一个问题：

```text
我们能不能从多本样本里抽出一个不换皮、能追读、可生产的小说生成机制。
```

如果这个闭环跑不通，不要继续扩写世界观。

## 5. 工作原则

```text
样本是显微镜。
提取是找证据。
归纳是找机制。
clean-room 是诊断。
盲审是暴露缺口。
内容内核是迁移对象。
house style 是我们的声音。
正式生产必须脱离单一样本。
```

更短一句：

```text
不借鉴一本书的形状。
不凭空写 AI 水文。
只融合抽象生成机制。
```

## 6. Critical-thinking 自问自答审查

这一节是对本文档本身的审查。按照 `critical-thinking-mode` 的要求，每次只追一个问题，直到追到根部假设。它不是最终结论，而是后续回修清单。

### Q1：为什么要做这套系统？

答：因为我们想要从 0 到 1 提升小说生产能力，而不是写低质量 AI 水文。

### Q1.1：为什么“直接让 AI 写”不够？

答：因为直接写通常只能得到通用套路：设定像、节奏像、开局像，但没有稳定的承诺、因果、阻力、翻牌和文风控制。

### Q1.2：为什么通用套路不能接受？

答：因为我们的目标不是产出一篇能读的草稿，而是能进入平台验证、能降低洗稿风险、能长期迭代的小说生产运行包。

### Q1.3：根部假设是什么？

答：根部假设是：

```text
小说生产能力不是靠单次生成获得，而是靠“样本证据 -> 机制归纳 -> clean-room 诊断 -> 回修”逐步长出来。
```

这个假设目前合理，但还没有被小说样本闭环验证，所以不能提前宣布系统成熟。

### Q2：为什么样本是显微镜，不是目标？

答：因为样本的价值是暴露机制，不是提供可复制内容。

### Q2.1：为什么不能把对标书当目标？

答：因为小说的高识别内容形状太多：主角身份、能力设定、组织架构、异常规则、桥段顺序、名场面、文风。一旦目标变成“像某本书”，很容易滑向换皮。

### Q2.2：为什么不能完全不借鉴？

答：因为完全不借鉴就会回到 AI 默认生成，缺少平台验证过的读者承诺、章节运动和追读机制。

### Q2.3：根部假设是什么？

答：根部假设是：

```text
可学习的是抽象生成机制，不可迁移的是高识别内容形状。
```

这个假设必须靠 `transfer_boundary` 和相似风险检查落地，否则只是口号。

### Q3：为什么要先做平台机制研究？

答：因为平台机制决定我们前期生产的是哪种文本。

### Q3.1：为什么不能先拆书？

答：如果不知道平台收益、推荐、签约、更新、违规和 AI 边界，拆出来的“好结构”可能不服务当前投稿策略。

### Q3.2：为什么平台机制不是小背景？

答：因为番茄、七猫、起点的核心验证不同。番茄更重直发、读完率、追读和免费流量；七猫更重内投、保底和编辑商业判断；起点更重付费追读、上架和长线订阅。它们会反过来改变前三章、前十章和第一卷设计。

### Q3.3：根部假设是什么？

答：根部假设是：

```text
小说生产文本不是纯创作物，也是平台机制下的商业文本。
```

这个假设成立，但要防止过度平台化，导致把番茄偏好误判成小说通用能力。

### Q4：为什么要分 L0/L1/L2/L3 样本？

答：因为不是所有样本都值得深拆。

### Q4.1：为什么不能对 30 本都深拆？

答：成本太高，而且会在还没判断路线价值前消耗大量时间。

### Q4.2：为什么不能只选 1-2 本爆款深拆？

答：单本爆款无法区分“路线机制”和“独特内容资产”。只拆一本，最容易洗成换皮。

### Q4.3：根部假设是什么？

答：根部假设是：

```text
样本分层能在成本、证据强度和相似风险之间取得平衡。
```

但这个假设要求我们必须有停止规则：L1 看不出 route 就不进 L2，L2 不支持机制就不进 runtime。

### Q5：为什么前 10 章是第一轮提取重点？

答：因为网文前期最能暴露读者承诺、主角发动机、阻力、兑现和章末钩子。

### Q5.1：为什么不是先看全书？

答：全书当然重要，但先看全书会拖慢验证。我们当前要判断方向能不能立项，前 10 章是最低成本窗口。

### Q5.2：为什么前 10 章不够？

答：前 10 章只能看开局留人，不能证明长篇可持续、第一卷可闭合、能力体系能支撑 50 万字。

### Q5.3：根部假设是什么？

答：根部假设是：

```text
前 10 章适合做路线候选判断，不适合做最终 route 归纳。
```

因此 P2 后必须有 P2.5，不能直接升级为 route model。

### Q6：为什么要有 route，而不是只按题材拆？

答：因为题材不决定小说怎么跑，route 才决定生成方式。

### Q6.1：为什么“都市高武”不够？

答：同样是都市高武，可以是系统升级、异常处置、规则破解、官方组织任务、破绽推演。它们的章节运动、爽点和阻力完全不同。

### Q6.2：为什么 route 也可能误导？

答：如果 route 名字只是我们发明的标签，没有样本证据支撑，它就会变成新的玄学词。

### Q6.3：根部假设是什么？

答：根部假设是：

```text
route 必须解释“主角怎么赢、阻力怎么反咬、读者为什么追”，否则不是 route。
```

### Q7：为什么要做 source-open 深读？

答：因为表格只能抓点，抓不到整段小说运动。

### Q7.1：为什么自动提取不够？

答：自动提取容易把章节拆成字段，但小说的追读往往来自连续运动：前一章的结果如何逼出后一章，信息如何逐步变形，阻力如何慢慢压住主角。

### Q7.2：为什么 source-open 也有风险？

答：人工通读容易变成读后感，或者被自己喜欢的桥段影响判断。

### Q7.3：根部假设是什么？

答：根部假设是：

```text
证据表和全文通读必须互相校正，任何一边单独使用都会失真。
```

### Q8：为什么要有内容内核卡？

答：因为 route 仍然太大，不能直接指导生成。

### Q8.1：为什么 route 不能直接变成 skill？

答：route 描述的是一类打法，但真正可迁移的是其中的最小成立机制，比如“异常规则如何制造误判、测试代价、破绽和反制”。

### Q8.2：为什么内容内核卡能降低洗稿风险？

答：因为它强迫我们只保留抽象循环和最小部件，同时列出 forbidden_content_shapes。

### Q8.3：根部假设是什么？

答：根部假设是：

```text
内容内核是 route 和正式生产之间的安全转换层。
```

如果没有这层，runtime skill 很容易偷渡单书桥段。

### Q9：为什么 clean-room 是诊断，不是生产？

答：因为它的作用是测试机制能不能迁移，而不是生成可直接发布正文。

### Q9.1：为什么不能把 clean-room 结果当正文？

答：clean-room 通常还没有完整 Story Bible、Style Bible、长期伏笔和平台投放策略，只能看机制是否初步跑通。

### Q9.2：为什么 clean-room 失败不能只加 prompt？

答：如果失败来自 route 错、因果断、阻力假、文风水，单纯加 prompt 会让系统越来越复杂，但问题根部没变。

### Q9.3：根部假设是什么？

答：根部假设是：

```text
生成失败首先是模型诊断信号，不是提示词优化任务。
```

### Q10：为什么必须有 house style？

答：因为“不像别人”不等于“像我们自己”。

### Q10.1：为什么文风不能最后随便补？

答：小说的阅读体验由叙述距离、句段节奏、对白气口、动作密度、心理比例、章末气口共同构成。文风如果只是最后润色，会和章节运动脱节。

### Q10.2：为什么文风也不能一开始就抽？

答：如果 route 和内容内核还没确定，先抽文风会得到“句子像了，剧情不动”的假成果。

### Q10.3：根部假设是什么？

答：根部假设是：

```text
Style Bible 应该在 route/内容内核之后、正式正文之前形成。
```

### Q11：为什么当前文档还不能算完整操作系统？

答：因为它有路线，但缺少执行模板和停止规则。

### Q11.1：缺模板会导致什么？

答：执行者会各写各的，证据粒度不一致，后续无法横向归纳。

### Q11.2：缺停止规则会导致什么？

答：系统会无限扩张：不断加样本、加字段、加理论，但不进入验证。

### Q11.3：根部假设是什么？

答：根部假设是：

```text
方法论只有落到模板、样本批次和停止条件，才会变成操作系统。
```

### Q12：下一步为什么不是继续完善大文档？

答：因为继续写理论的边际收益已经下降。

### Q12.1：那下一步应该验证什么？

答：验证 P2 前 10 章快拆字段是否够用，能不能支撑 route_candidate 判断。

### Q12.2：为什么先做 3 本校准批次？

答：3 本足够暴露字段缺口，又不会在错误字段上浪费太多样本。

### Q12.3：根部结论是什么？

```text
当前文档可以作为路线图。
但下一步应补模板并跑 3 本校准批次。
如果校准批次无法产出 route_candidate，说明提取协议要回修，而不是继续扩写方法论。
```

## 7. 策略漏洞循环

这一节专门回答：我对这个策略有没有 100% 信心？

结论：

```text
没有事实上的 100% 信心。
```

原因：

```text
文档策略只有经过样本校准、clean-room 诊断、盲审和回修后，才可能接近事实置信。
当前最多只能说：已识别主要高风险漏洞，并为每个漏洞设置修复措施和验证门。
```

### 循环 1：归纳颗粒度风险

问题：

```text
短视频可以按单条视频归纳，小说如果没有多颗粒度矩阵，会把“章节、事件链、第一卷、整本书”的功能混在一起。
```

影响：

```text
前 10 章的开局规律可能被误判成整本书规律。
单章节奏可能被误判成 route。
整本书长线可能过早压死开局验证。
```

修复：

```text
已新增 B. 小说按什么维度归纳和诊断。
明确单章、前3章、前10章、前30章、事件链、第一卷、上架前后、整本书各自用途。
```

验证门：

```text
P2 只允许产出 route_candidate。
P3 才能验证事件链。
P4 才能归纳 route。
任何前10章结论不得直接升级为整本书规律。
```

剩余风险：

```text
中。需要 3 本校准样本验证颗粒度是否够用。
```

### 循环 2：模板缺失风险

问题：

```text
只有方法论，没有可填写模板，会导致每次提取粒度不一致。
```

影响：

```text
证据无法横向比较。
route_candidate 判断会变成主观读后感。
后续无法做稳定回修。
```

修复：

```text
新增 schemas/sample_record.yaml。
新增 schemas/chapter_evidence.yaml。
新增 schemas/content_kernel_card.md。
新增 schemas/blind_review_report.md。
```

验证门：

```text
3 本 P2 校准样本必须全部使用同一套模板。
字段无法填写、重复填写、无决策价值的字段必须回修。
```

剩余风险：

```text
中。模板要经过第一轮样本校准才能降风险。
```

### 循环 3：样本偏差风险

问题：

```text
如果只选我们熟悉或喜欢的书，归纳出的不是平台/赛道机制，而是我们的阅读偏好。
```

影响：

```text
方向判断失真。
route 过拟合。
clean-room 可能只适合我们自己看，不适合平台读者。
```

修复：

```text
L0 样本池必须包含成功样本、普通样本、失败/争议样本。
样本记录必须写 possible_bias。
番茄是第一训练场，七猫和起点作为平台对照。
```

验证门：

```text
进入 L2 的样本必须说明为什么代表该 route。
至少保留 1 个反例或失败样本做对照。
```

剩余风险：

```text
中高。必须等样本池实际使用后才能降低。
```

### 循环 4：共性误判风险

问题：

```text
多本书都出现的东西不一定是有效机制，可能只是题材公约、平台模板或低质套路。
```

影响：

```text
把平庸共性写进运行包。
生成稿像套路合集。
```

修复：

```text
共性只进入 candidate，不直接进入 runtime。
route 横向归纳必须写 counterexamples 和 failure_modes。
内容内核卡必须回答 why_it_works，不只写 what_is_common。
```

验证门：

```text
一条机制必须能解释成功样本和失败样本的差异。
不能解释差异的共性，不进入 runtime。
```

剩余风险：

```text
中。需要 route 归纳时强制执行。
```

### 循环 5：换皮/洗稿风险

问题：

```text
小说的可识别资产不只在句子，还在人设、能力、组织、事件顺序、异常规则和名场面。
```

影响：

```text
硬风险：平台处罚、侵权、签约受阻。
软风险：读者说像、作者圈质疑、评论区带节奏。
```

修复：

```text
每条证据必须有 transfer_boundary。
内容内核卡必须有 forbidden_content_shapes。
clean-room 写手禁止读取原书全文、桥段顺序、角色/能力/组织/异常规则。
```

验证门：

```text
clean-room 后做相似风险审查。
任何单书高识别内容形状出现，退回重写。
```

剩余风险：

```text
中高。必须结合平台规则、法规和人工审稿持续压低。
```

### 循环 6：文风后置风险

问题：

```text
如果只归纳结构和内核，文风最后随便补，会变成“结构对但读起来像 AI”。
```

影响：

```text
读者弃书。
编辑判断低质。
平台风控可能识别为套路化/水文。
```

修复：

```text
文风不在 P2 最先抽，但必须在 clean-room 后、正式正文前形成 Style Bible。
Style 诊断作为独立诊断颗粒度。
```

验证门：

```text
同一场景至少做一次 Style 诊断。
Style Bible 必须能约束叙述距离、句段节奏、对白、动作、心理、环境、章末气口。
```

剩余风险：

```text
中。后续必须补 Style Bible 模板。
```

### 循环 7：clean-room 误用风险

问题：

```text
clean-room 可能被误用成正式生产，或者被当成“像不像样本”的测试。
```

影响：

```text
过早写正文。
盲审方向错误。
系统滑回仿作。
```

修复：

```text
文档已明确 clean-room 是诊断，不是生产。
盲审问题改成追读、自嗨、AI味、相似风险、下一章压力。
```

验证门：

```text
clean-room 产物不得直接发布。
clean-room 失败必须先定位 route/因果/阻力/翻牌/文风/相似风险层级。
```

剩余风险：

```text
低到中。执行时仍需审查。
```

### 循环 8：过度平台化风险

问题：

```text
番茄是训练场，但如果所有判断都按番茄来，会牺牲小说通用能力。
```

影响：

```text
运行包变成番茄专项技巧，而不是小说生产能力。
```

修复：

```text
番茄样本验证免费流量/读完率/追读。
七猫对照编辑商业判断。
起点对照付费追读和长线可信度。
```

验证门：

```text
进入正式运行包的机制必须标注 platform_fit。
只适用于番茄的机制不能写成小说通用机制。
```

剩余风险：

```text
中。需要后续七猫/起点对照样本。
```

### 循环 9：停止规则不足风险

问题：

```text
没有停止规则，系统会无限扩张。
```

影响：

```text
一直研究，不写书。
一直加字段，不验证。
```

修复：

```text
新增停止规则：
P2 无法产出 route_candidate，停止深拆，回修提取协议。
L2 无法解释事件链，停止 route 归纳。
route 只有一本书支撑，停止升级 runtime。
clean-room 连续两轮无追读，停止扩写世界观。
相似风险无法降到可控，停止该 route。
```

验证门：

```text
每阶段必须有 continue / revise / stop 决策。
没有决策不得进入下一阶段。
```

剩余风险：

```text
低到中。需要写进模板执行。
```

### 事实置信结论

当前不能说 100%。

更准确的状态是：

```text
策略层高风险漏洞已识别。
已为每个高风险漏洞设置修复措施和验证门。
剩余事实置信必须通过 3 本 P2 校准批次获得。
```

下一轮循环：

```text
1. 用新增模板跑 3 本 P2 前10章快拆。
2. 检查能否产出 route_candidate。
3. 如果不能，回修模板和提取协议。
4. 如果能，再进入 P3 事件链深读。
```
