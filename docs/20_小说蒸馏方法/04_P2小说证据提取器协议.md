# P2 小说证据提取器协议

## 0. 定位

这个文件是 P2/L1 前 10 章结构证据抽取的操作手册。

它不是拆书读后感，也不是泛方法论。它规定：

```text
输入什么
提取什么
每个字段怎么判断
有效证据长什么样
无效证据长什么样
AI 抽取和 source-open 校准怎么分工
什么证据能进入 P2.5 路线候选判断
```

P2 的目标不是证明一条路线成立，而是给 P2.5 提供可核对证据。

## 1. P2 输入包

每本样本进入 P2 前，必须先准备输入包。

### 1.1 必备输入

| 字段 | 说明 |
| --- | --- |
| book_id | L0 样本编号 |
| title | 书名 |
| author | 作者 |
| platform | 平台 |
| source_url | 来源链接 |
| sampled_at | 采样日期 |
| tags | 平台标签 |
| intro | 简介 |
| chapter_scope | 实际读取章节，默认前 10 章 |
| text_access | 章节文本、章节摘要、人工阅读记录或可回看链接 |
| sample_risk | 样本风险 |
| platform_submission_path | 样本更像直发验证、编辑内投、付费长线还是跨平台候选 |
| platform_monetization_vector | 样本可能适配的收益来源 |
| reader_feedback_scope | 是否采集评论、评分、书评区、章评或推书反馈 |

没有 `source_url`、`sampled_at`、`chapter_scope` 的样本，不进入 P2。

### 1.2 文本来源原则

允许：

1. 人工阅读后的结构摘要。
2. 合法可访问页面的章节标题和简短结构记录。
3. 自己记录的章节运动表。
4. 不复制大段原文的局部证据摘要。

禁止：

1. 把大体量小说原文提交进仓库。
2. 复制完整章节。
3. 复制高识别段落作为训练材料。
4. 把样本文字直接喂给后续生成环节。

### 1.3 评论来源原则

评论、书评、章评、推书帖可以采集，但它们只属于“读者反应证据”，不是剧情素材。

允许：

1. 统计读者反复提到的爽点、弃点、看不懂点、追更理由。
2. 记录读者对标签、简介、开局、节奏、主角能力、反派压迫、翻牌的反应。
3. 记录读者是否认为作品像某本书、AI味重、套路化或水文。
4. 用短摘要记录，不复制大段评论原文。

禁止：

1. 把读者评论中的桥段建议直接写进我们的项目。
2. 复制评论区里的设定脑洞、人物设计、能力方案或剧情方案。
3. 把单条高赞评论当成全体读者结论。
4. 为了迎合评论牺牲样本正文证据和平台规则。

## 2. P2 输出包

每本样本输出一个 `book_p2_evidence_pack`。

最低包含 6 类证据：

```text
E1 reader_promise_evidence
E2 chapter_motion_evidence
E3 plot_causality_evidence
E4 opposition_evidence
E5 reveal_flip_evidence
E6 support_function_evidence_mvp
```

可选包含：

```text
E7 mvp_renderer_evidence
E8 adaptation_evidence_mvp
E9 reader_feedback_evidence
E10 editor_outline_gate_evidence
```

P2 阶段不做完整 Style Bible，不做完整人物关系网，不做完整世界观。

但 P2 必须开始记录“编辑门”和“大纲学习”信号。原因是：七猫等平台的投稿包需要大纲，起点/阅文看长期主线可信度，番茄在 2万/5万/8万字签约节点评估作品质量。前 10 章不能只判断“好不好看”，还要判断“是否有可持续连载和可审稿材料的苗头”。

## 3. 通用证据卡格式

所有 evidence 必须有通用字段。

```yaml
evidence_id:
book_id:
evidence_type:
location:
source_channel:
source_basis:
summary:
structural_function:
changed_state:
platform_strategy_note:
reader_reaction_note:
confidence:
transfer_boundary:
risk_note:
extractor_note:
source_open_status:
```

### 字段解释

| 字段 | 填写规则 |
| --- | --- |
| evidence_id | 如 `S010-E3-002` |
| book_id | 对应 L0 样本编号 |
| evidence_type | E1-E10 |
| location | 章节、简介、书名、章末等位置 |
| source_channel | text / metadata / comment / forum / platform_rule |
| source_basis | 证据来自简介、章节摘要、人工阅读记录还是原网页 |
| summary | 不复制原文，只做结构摘要 |
| structural_function | 它服务承诺、因果、阻力、翻牌、节奏还是文风 |
| changed_state | 这条证据是否改变主角处境、读者预期或故事压力 |
| platform_strategy_note | 它对签约、推荐、收益、更新成本或违规风险有什么影响 |
| reader_reaction_note | 如果来自评论/反馈，记录读者实际反应；正文证据可填 none |
| confidence | high / medium / low |
| transfer_boundary | 迁移边界标签 |
| risk_note | 相似风险、平台风险或软风险说明 |
| extractor_note | 提取者备注，不写主观读后感 |
| source_open_status | pending / checked / revised |

### confidence 判断

| 等级 | 标准 |
| --- | --- |
| high | 能定位章节，结构功能明确，删除后会影响后续 |
| medium | 能定位章节，但功能需要后续 source-open 校准 |
| low | 只有简介/标签/模糊摘要支撑，暂不进入归纳 |

### transfer_boundary 标签

必须使用 P0.5 的标签：

```text
transferable_mechanism
genre_convention_rebuild
observe_only
prohibited_shape
legal_platform_review
soft_risk_review
```

没有 `transfer_boundary` 的证据不能进入 P2.5。

### platform_strategy_note 判断

P2 不是只判断“这段写得好不好”，还要判断它服务哪个平台压力。

可记录：

```text
signing_review
recommendation_evaluation
editor_submission_review
retention_data
vip_conversion
full_attendance
activity_bonus
ip_adaptation
compliance_review
```

示例：

```yaml
platform_strategy_note: "第1章300字内出主角、危机和背景，明显服务番茄签约评估中对开篇、代入感、主线和主角推动力的要求。"
```

### editor_review_signal 判断

编辑门不等于最终审稿，但 P2 每章要记录编辑视角信号。

可记录：

```text
marketable_hook
clear_mainline
protagonist_drive
smooth_reading
conflict_visible
logic_gap
weak_intro
too_many_terms
generic_ai_prose
similarity_alarm
```

示例：

```yaml
editor_review_signal: "第1章能在开头建立危机和主角目标，但书名/简介承诺的异常处理卖点尚未出现；编辑视角会继续看，但需要第2-3章尽快展示核心卖点。"
```

### outline_signal 判断

大纲学习不是现在写完整大纲，而是从样本前 10 章反推它是否有长线结构能力。

可记录：

```text
first_volume_direction
long_term_pressure
resource_cycle
relationship_cycle
organization_thread
hidden_big_problem
no_outline_signal
sustainability_risk
```

示例：

```yaml
outline_signal: "第8章出现组织权限和异常档案库，说明后续可以通过任务、权限和未解档案扩展第一卷，不只是单次事件。"
```

### ai_quality_signal 判断

AI 低质风险不是“用了 AI”本身，而是文本呈现出批量生成、逻辑拼凑或人物工具化。

可记录：

```text
generic_expression
empty_emotion
logic_jump
repeated_scene_move
character_inconsistency
setting_dump
water_content
human_control_visible
```

示例：

```yaml
ai_quality_signal: "连续两章用同样的旁白总结主角震惊和世界危险，但没有新行动和状态变化，存在水文/AI感风险。"
```

## 4. E1 Reader Promise Evidence

### 解决什么

判断这本书对读者承诺什么，以及前 10 章是否兑现。

### 必填字段

| 字段 | 判断方式 |
| --- | --- |
| pre_click_promise | 书名、简介、标签承诺什么 |
| opening_promise | 第 1 章实际给什么 |
| promise_type | 爽、悬疑、成长、身份、压迫、智斗、关系、世界观 |
| main_or_side | 主承诺 / 副承诺 / 阶段承诺 |
| carrier | 承诺载体：危机、能力、身份、案件、组织、关系 |
| first_payoff_chapter | 第一次兑现章节 |
| payoff_action | 兑现时主角或局势发生什么动作 |
| payoff_result | 读者获得什么满足 |
| promise_drift | 简介承诺和正文是否偏离 |

### 有效证据

```text
第 4 章，主角第一次用“读档后保留信息”避开死亡，并反推出规则漏洞。它兑现的是“靠信息差破局”的承诺，不是单纯战力升级。
```

### 无效证据

```text
这本书是智斗爽文。
```

问题：没有位置、没有兑现动作、没有读者满足。

### 迁移边界判断

| 情况 | 标签 |
| --- | --- |
| “前 3 章压迫，第 5 章小破局”这种节奏功能 | transferable_mechanism |
| 系统、觉醒、退婚、官方组织这类通用承诺载体 | genre_convention_rebuild |
| 某本书独特书名、简介钩子、专属能力承诺 | observe_only 或 prohibited_shape |

## 5. E2 Chapter Motion Evidence

### 解决什么

判断前 10 章每章是否让故事向前走。

### 每章必填字段

| 字段 | 判断方式 |
| --- | --- |
| chapter | 章节 |
| start_state | 本章开局主角处境 |
| trigger | 什么事件启动本章 |
| protagonist_goal | 主角本章想要什么 |
| conflict | 谁/什么阻止他 |
| action | 主角做了什么 |
| new_information | 读者或主角知道了什么新信息 |
| payoff | 本章兑现了什么 |
| end_hook | 章末留下什么压力 |
| changed_state | 本章结束后处境如何变化 |

### 有效证据

```text
第 2 章开局主角被卷入异常现场；触发是队友失联；目标是逃出现场；冲突是出口规则失效；行动是测试三条出口路径；结尾发现“安全出口”只对死人开放，逃生目标变成寻找活人规则。
```

### 无效证据

```text
第 2 章继续铺世界观，节奏还行。
```

问题：没有目标、冲突、行动、状态变化。

### changed_state 判断

必须至少满足一项：

1. 主角处境变化。
2. 主角目标变化。
3. 主角风险升级。
4. 读者疑问升级。
5. 关系或资源发生变化。

如果都没有，这章不能作为高价值章节运动证据。

## 6. E3 Plot Causality Evidence

### 解决什么

判断剧情是不是硬接。

### 必填字段

| 字段 | 判断方式 |
| --- | --- |
| previous_consequence | 上一幕留下什么后果 |
| forced_action | 主角因此必须做什么 |
| why_now | 为什么现在必须做 |
| obstacle | 阻碍是什么 |
| action | 主角实际行动 |
| result | 行动结果 |
| cost | 代价 |
| next_pressure | 这个结果如何逼出下一幕 |
| removable_alarm | 删除该场景后后文是否仍成立 |

### 有效证据

```text
因为第 3 章主角暴露了异常感知能力，组织在第 4 章锁定他身份；他不能再装普通人，只能接受临时协助任务；接受任务后获得权限，但也被纳入监控，后续行动受限。
```

### 无效证据

```text
主角接到任务，所以去调查。
```

问题：没有上一幕后果，没有“不能不做”的压力。

### 因果强度

| 等级 | 标准 |
| --- | --- |
| strong | A 不发生，B 就不会发生 |
| medium | A 推动 B，但 B 也可能由别的原因发生 |
| weak | A 和 B 只是相邻发生 |

只有 `strong` 和部分 `medium` 能进入 P2.5。

## 7. E4 Opposition Evidence

### 解决什么

判断阻力是否会主动改变主角处境。

### 必填字段

| 字段 | 判断方式 |
| --- | --- |
| opposition_type | 人物、组织、规则、信息差、资源、代价、关系、盲区 |
| opposition_actor_or_rule | 具体阻力是谁或什么规则 |
| opposition_goal | 阻力想要什么 |
| leverage | 阻力筹码 |
| active_move | 阻力主动做了什么 |
| protagonist_response | 主角如何应对 |
| cost_to_protagonist | 主角付出什么 |
| backlash | 阻力如何反咬 |
| escalation | 是否升级 |
| repeatability | 能否持续生产剧情 |

### 有效证据

```text
组织不是只发任务，而是在第 6 章用保密协议限制主角报警和求助；主角获得调查权限，但失去公开解释能力。这个组织阻力能持续制造“不能说真相”的压力。
```

### 无效证据

```text
组织很神秘。
```

问题：没有目标、筹码、主动动作、反咬。

### 阻力有效性判断

有效阻力必须至少满足两项：

1. 有目标。
2. 有筹码。
3. 会主动行动。
4. 让主角付出代价。
5. 能反咬。
6. 可复用。

## 8. E5 Reveal / Flip Evidence

### 解决什么

判断信息差和翻牌是否推动追读。

### 必填字段

| 字段 | 判断方式 |
| --- | --- |
| open_question | 打开什么疑问 |
| hidden_info | 隐藏什么信息 |
| clue | 提前给了什么线索 |
| false_explanation | 临时误导解释是什么 |
| reader_assumption | 读者可能以为什么 |
| reveal_chapter | 第几章揭示 |
| flip_effect | 揭示后局势如何变化 |
| payoff | 回收什么 |
| next_question | 翻牌后打开什么新问题 |

### 有效证据

```text
前 3 章一直让读者以为异常规则是“不能回头”；第 5 章翻牌发现真正规则是“不能被镜中人提前看到下一步”。翻牌让主角原逃生方案失效，必须换成遮蔽视线的解法。
```

### 无效证据

```text
第 5 章有反转。
```

问题：没有线索、误导、局势变化。

### 强翻牌标准

强翻牌必须改变至少一项：

1. 主角目标。
2. 解法。
3. 敌我关系。
4. 风险等级。
5. 世界规则。
6. 下一章压力。

只解释过去、不改变未来，是弱翻牌。

## 9. E6 Support Function Evidence MVP

### 解决什么

判断能力、组织、民俗、地域、怪物、道具是否服务剧情，而不是设定展示。

P2 只做 MVP，不做完整设定体系。

### 必填字段

| 字段 | 判断方式 |
| --- | --- |
| support_item | 能力、组织、民俗、地名、案件、道具、传说 |
| first_appearance | 首次出现位置 |
| function | 制造行动、限制、线索、误导、代价、视觉场面 |
| plot_dependency | 后文是否依赖 |
| self_indulgence_risk | 是否只是设定展示 |

### 有效证据

```text
“异常管理局”首次出现不是为了设定世界观，而是限制主角公开报警、提供调查权限，并制造保密代价。
```

### 无效证据

```text
这个组织设定很酷。
```

问题：没有剧情功能。

## 10. E7 MVP Renderer Evidence

### 解决什么

P2 不做完整文风，但 clean-room 前 3 章需要最小叙述约束，防止生成测试被语言质量污染。

### 只抽 5 个字段

| 字段 | 判断方式 |
| --- | --- |
| pov_distance | 叙述离主角有多近 |
| dialogue_action_ratio | 对白和动作比例 |
| exposition_method | 设定如何进入行动 |
| ending_breath | 章末怎么收钩 |
| ai_smell_alarm | 哪些表达显得空泛或 AI |

### 有效证据

```text
章末不总结情绪，而是落在一个未解决动作或新信息上，例如“门外传来第二个自己的声音”。这说明该类样本章末气口偏行动钩/信息钩。
```

### 无效证据

```text
文笔不错。
```

问题：不能指导生成。

## 11. E8 Adaptation Evidence MVP

### 解决什么

判断是否有漫画、短剧、AI 漫改潜力，但不能压过小说追读。

### 必填字段

| 字段 | 判断方式 |
| --- | --- |
| visual_scene | 强视觉场景 |
| anomaly_visualization | 异常/能力如何可视化 |
| character_tag | 角色标签是否一眼能懂 |
| episode_hook | 是否可切成单集钩子 |
| production_risk | 是否成本过高 |

### 有效证据

```text
“全校宿舍灯同时熄灭，只剩主角房间镜子发光”具备低成本视觉钩子，能作为漫画/短剧单集开场。
```

### 无效证据

```text
这个设定适合短剧。
```

问题：没有具体视觉、镜头或单集钩子。

## 12. E9 Reader Feedback Evidence

### 解决什么

判断读者真实反应是否支持正文证据。

E9 不是必做项。只有样本评论量足够、可公开查看、且不需要复制大段评论时才采集。

### 必填字段

| 字段 | 判断方式 |
| --- | --- |
| feedback_scope | 书评区、章评、评分页、推书帖、论坛讨论 |
| feedback_sample_size | 粗略样本量，例如20条、100条、热门前30条 |
| praised_points | 读者反复夸什么 |
| drop_points | 读者反复吐槽或弃书点 |
| confusion_points | 读者看不懂、误解或争议点 |
| chase_reason | 读者为什么追下一章 |
| similarity_alarm | 是否有人说像某本书、融梗、AI味、水文 |
| related_text_evidence | 它对应哪条 E1-E8 正文证据 |
| actionability | 能转化成哪个抽象机制或风险门 |

### 有效证据

```text
热门章评中多次提到“主角不是硬刚，而是抓住规则漏洞反杀”，可回连 E5-003 的翻牌证据。说明读者实际爽点来自“破绽识别 + 后手兑现”，不是单纯战力碾压。
```

### 无效证据

```text
评论都说好看，所以这本可以学。
```

问题：没有说明好看在哪里，也不能回到正文机制。

### 使用边界

1. E9 只能验证读者反应，不能单独生成 route。
2. E9 必须回连至少一条 E1-E8 正文证据，否则只能作为 `observe_only`。
3. 涉及相似、融梗、AI味、水文的评论，优先进入 `soft_risk_review`。
4. 高赞评论里的原创脑洞、剧情建议、设定方案禁止迁移。

## 13. E10 Editor / Outline Gate Evidence

### 解决什么

判断样本前 10 章是否具备编辑视角的签约/投稿/直发验证潜力，以及是否透露出可学习的大纲能力。

这个证据不要求现在写完整大纲，但必须判断：

```text
开篇是不是能过编辑第一眼？
前 10 章是不是有小闭环？
是否出现第一卷方向？
是否有长期压力、资源循环、关系循环或未解核心问题？
是否存在 AI 低质、水文、相似或平台风险？
```

### 必填字段

| 字段 | 判断方式 |
| --- | --- |
| target_platform | 番茄 / 七猫 / 起点 / unknown |
| editor_package_signal | 书名、简介、标签、开篇是否形成可审材料 |
| opening_signing_potential | 前 3 章是否有签约/直发验证潜力 |
| first_10_outline_signal | 前 10 章是否能反推出第一卷方向 |
| long_term_pressure | 后续 30万字以上靠什么继续 |
| sustainability_risk | 是否只有开头噱头，长线乏力 |
| ai_low_quality_risk | 是否像 AI 水文、批量生成、逻辑拼凑 |
| similarity_or_plagiarism_risk | 是否像某本书的桥段、能力、组织、文风 |
| editor_gate_decision | proceed / revise / stop / insufficient_evidence |

### 有效证据

```text
样本前 3 章完成危机、主角目标和能力展示，第 8 章引入组织档案库，说明后续可以通过任务、权限和未解案件扩展第一卷；但第 1 章书名承诺和正文卖点错位，编辑门判断为 revise。
```

### 无效证据

```text
这本编辑应该会喜欢。
```

问题：没有平台、没有开篇信号、没有大纲能力、没有风险判断。

### 和 novel-editor-gate skill 的关系

P2 填表时可以调用：

```text
.agents/skills/novel-editor-gate/SKILL.md
```

该 skill 负责给出 editor gate 判断；本协议负责把判断落到证据字段里。

## 14. AI 抽取与 source-open 校准分工

### 14.1 AI 抽取负责什么

AI 可以先做：

1. 按字段生成初版证据卡。
2. 标出可能的章节运动。
3. 标出疑似因果链。
4. 标出阻力和翻牌候选。
5. 初步标注 transfer_boundary。
6. 初步写 risk_note。
7. 初步聚类评论反馈，但不能复制评论原文。

AI 不能直接做：

1. 把低置信证据升级为结论。
2. 判定正式 route。
3. 判定无侵权风险。
4. 把样本文字迁移进生成稿。
5. 把评论当作路人评价，而不回到正文证据。

### 14.2 source-open 校准负责什么

source-open 必须校准：

1. 证据是否真的存在。
2. 章节位置是否准确。
3. AI 是否把评价当证据。
4. 因果链是否硬接。
5. transfer_boundary 是否过于宽松。
6. risk_note 是否遗漏高识别内容形状。
7. 评论反馈是否有样本量、是否偏向极端读者、是否能回连正文证据。

### 14.3 校准状态

| 状态 | 含义 |
| --- | --- |
| pending | AI 初抽，未校准 |
| checked | 人工或 source-open 校准通过 |
| revised | 校准后修改 |
| rejected | 删除，不进入归纳 |

## 15. P2 工作流

每本书按以下顺序处理：

```text
1. 准备输入包
2. 读书名/简介/标签，抽 E1 初版
3. 逐章做 E2 章节运动表
4. 从 E2 中挑关键场景抽 E3 因果链
5. 从 E2/E3 中抽 E4 阻力
6. 从疑问和反转中抽 E5 翻牌
7. 抽 E6 支撑物功能 MVP
8. 如果准备 clean-room，再抽 E7 MVP 文风
9. 如有强视觉场景，抽 E8 改编 MVP
10. 如评论/书评/章评可采集，抽 E9 读者反馈
11. 调用或模拟 novel-editor-gate，抽 E10 编辑/大纲门证据
12. 标注 platform_strategy_note、editor_review_signal、outline_signal、ai_quality_signal
13. 标注 transfer_boundary 和 risk_note
14. source-open 校准
15. 输出 book_p2_evidence_pack
```

## 16. 一条证据进入 P2.5 的门槛

证据必须同时满足：

1. 能定位到书名/简介/章节/场景。
2. 有明确结构功能。
3. 能说明状态变化。
4. 不是单纯评价。
5. 有 `transfer_boundary`。
6. 有 `risk_note`。
7. 能说明平台适配或明确标记为 unknown。
8. `confidence` 不是 low。
9. `source_open_status` 是 checked 或 revised。

以下证据不能进入 P2.5：

1. 只来自模糊印象。
2. 只有标签，没有章节支撑。
3. 是内容形状而不是机制。
4. transfer_boundary 是 `prohibited_shape`。
5. transfer_boundary 是 `legal_platform_review` 且未复核。
6. transfer_boundary 是 `soft_risk_review` 且未改写。

## 17. P2.5 输出依赖

P2 完成后，P2.5 才能判断路线候选。

P2.5 只看这些：

```text
checked/revised evidence
transferable_mechanism
genre_convention_rebuild
medium/high confidence
明确 risk_note
```

P2.5 不看这些：

```text
prohibited_shape
observe_only 的内容形状
未经校准的 AI 结论
低置信评价
单一样本高识别元素
```

## 18. 最小闭环版本

为了避免工作量爆炸，第一轮 P2 校准批次只做：

```text
3 本样本
每本前 10 章
E1-E6 必做
E7 只做 MVP
E8 只记录强视觉钩子
E9 只在评论可采集时做反馈验证
E10 只做编辑门和大纲学习信号，不写完整大纲
```

第一轮不做：

```text
完整 Story Bible
完整 Style Bible
完整第一卷深拆
完整人物关系网
完整能力体系
```

第一轮的目的只有一个：

```text
验证这个提取器能不能产出足够支撑 P2.5 的证据。
```

如果第一轮证据无法支撑路线候选判断，说明提取器协议要回修，而不是继续加样本。
