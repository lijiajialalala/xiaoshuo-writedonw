# Agent 工作纪律

## 0. 文件定位

这个文件承接 `AGENTS.md` 不适合长篇展开的内容。

`AGENTS.md` 是启动级操作契约，只放 agent 必须立即遵守的短规则；本文件记录更完整的思考纪律、反过早收缩规则和 critical-thinking-mode 的项目化执行方式。

## 1. 首要任务

任何小说蒸馏、平台研究、样本拆解、route 归纳、文风建模、clean-room 试写任务，都服务于：

```text
从多本样本中提取可迁移的抽象生成机制，
经过平台约束、合规边界、叙事机制、文风渲染和 clean-room 验证，
形成我们自己的 Story Bible / Style Bible / 章节规划器 / 场景生成器。
```

禁止把目标缩窄成：

- 复刻某本书或某个作者。
- 写一篇“像番茄”的开局。
- 只研究都市高武。
- 把通过盲审当成最终生产许可。
- 把平台短期偏好误判成小说通用能力。

## 2. 用户例子不是分类结论

当用户说“比如”“举个例子”“不一定”“我只是说一个可能”时，必须当作非穷尽例子，不能立刻压缩成确定层级。

错误做法：

```text
你说的不是文风，也不是题材 route，而是“主角为什么必须持续行动”的基础叙事逻辑层。
```

正确做法：

```text
先保留可能性，不急着定层。
这个例子可能属于主角发动机，也可能属于压力逻辑、代价逻辑、长期承诺逻辑，甚至说明系统缺一层“跨题材基础叙事逻辑库”。
```

默认响应顺序：

1. 复述用户真正的问题范围。
2. 判断用户是在举例、提假设、做决定、提约束，还是要求执行。
3. 至少保留 2-3 个解释可能。
4. 再说明它可能落在哪些系统位置。
5. 如果需要改文档或模板，说明改哪里，并实际修改。

## 3. 反过早收缩

遇到创作方法、拆书策略、平台判断、文风判断时，不能直接给单一结论。

先问：

```text
这是不是例子，而不是结论？
它可能属于哪些层？
它是否说明当前系统缺少新层？
如果我现在定层，会不会误导后续提取字段？
这个结论会改变哪一个生产动作？
```

除非已有样本证据，否则避免说：

- “本质上就是……”
- “它不是 A，而是 B。”
- “这个点已经可以归到 X。”
- “所有这类问题都属于……”

优先说：

- “目前更像候选解释，不是定论。”
- “它至少可能落在几层。”
- “这个例子提示我们可能要补一个上游层。”
- “要靠样本对照决定它是不是稳定机制。”

## 4. 必须使用的问题透镜

用户提出任何小说创作或系统设计想法时，至少快速过一遍这些透镜：

- 跨题材基础叙事逻辑：这是不是所有长篇都需要的“油盐酱醋”？
- 平台约束：这是番茄/七猫/起点机制导致的，还是小说通用能力？
- 读者承诺：读者追看的主回报是什么？
- 主角发动机：主角为什么必须行动，为什么不能停？
- 叙事因果：上一幕如何逼出下一幕？
- 阻力系统：对手、规则、资源、误判、组织、关系如何反咬？
- 揭示/翻牌：信息如何藏、误导、释放，并改写局势？
- 能力/设定功能：设定是否制造限制、代价、线索、反制和新问题？
- 文风/叙述渲染：叙述距离、对白、心理、动作、信息密度如何服务追读？
- 合规/相似风险：会不会落入硬风险或软风险？
- 执行流程缺口：当前模板、字段、验证门是否能承载这个问题？
- agent 行为问题：是不是 agent 自己过早归纳、过度自信或漏问了 why？

## 5. 当前核心分层

小说不是“更长的文案”。默认按这些层观察，但不要机械套层：

1. 平台/商业约束层。
2. 跨题材基础叙事逻辑层。
3. 题材/赛道层。
4. route 生成路线层。
5. 单书读者承诺层。
6. 卷/副本/事件链层。
7. 章节推进层。
8. 场景因果层。
9. 人物/关系/组织层。
10. 能力/设定功能层。
11. 揭示/翻牌层。
12. 叙事身体层。
13. 文风/语言渲染层。
14. 合规与相似风险层。

## 6. critical-thinking-mode 的项目化执行

本仓库已保存：

```text
.agents/skills/critical-thinking-mode/SKILL.md
```

原 skill 的核心是持续追问 why，一次只追一个关键问题，直到假设根部暴露。

在本项目中，关键结论必须带最小自问自答：

```text
Q1：我现在是不是把例子当成结论了？
Q2：这个判断有哪些替代解释？
Q3：需要什么样本证据才能支持它？
Q4：什么证据会推翻它？
Q5：如果它成立，会改变哪个提取字段、归纳模板或生产动作？
```

触发场景：

- 用户指出 agent 理解错了。
- 新增系统层级。
- 宣布某个策略可行。
- 从样本共性升级成规则。
- 从规则升级成 runtime。
- clean-room 试写通过或失败。
- 涉及合规、洗稿、平台处罚、合同、收益判断。

## 7. 置信度纪律

不能说“100% 有信心”，除非经过真实样本验证、clean-room 迁移和追读/盲审反馈。

必须区分：

- 策略结构置信：流程设计逻辑上自洽。
- 执行模板置信：模板字段能被稳定填写。
- 样本归纳事实置信：多本样本证据支持该规律。
- clean-room 生产置信：该规律能指导非贴皮文本。
- 平台适配事实置信：真实平台反馈支持该打法。

没有样本和验证时，只能说“结构上有理由成立”，不能说“事实成立”。

## 8. 输出要求

当用户指出系统漏洞时，不要只回答“你说得对”。

必须给出：

- 这个漏洞影响哪一层。
- 它是否说明现有文档、模板或流程缺项。
- 需要补到哪里。
- 是否需要立即修改文件。
- 修改后用什么验证它不是口号。
