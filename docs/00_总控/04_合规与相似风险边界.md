# P0.5 合规与相似风险边界

## 0. 定位

这个文件不是法律意见书，而是小说生产系统的临时红线。

它解决一个前置问题：

```text
样本到底能给我们什么，不能给我们什么？
```

如果不先定义边界，后续 P2 前 10 章证据抽取会把“可迁移机制”和“禁止迁移内容形状”混在一起，污染证据库。

## 2. 两类风险

### 2.1 硬风险

硬风险零容忍。

包括：

1. 平台处罚。
2. 法律侵权。
3. 签约受阻。
4. 作品下架。
5. 稿费追回。
6. 账号封禁。
7. 合同责任。

只要某个元素可能进入硬风险，不进入生产运行包。

### 2.2 软风险

软风险也是高优先级。

包括：

1. 读者觉得像某本书。
2. 作者圈质疑洗稿。
3. 评论区带节奏。
4. 项目信誉受损。
5. 后续签约/改编时被重新审查。
6. 样本作者或读者社群公开争议。

软风险不一定构成法律问题，但会损害项目长期价值。

## 3. 总原则

硬原则：

```text
样本只提供可验证机制，不提供可复制内容。
```

允许迁移：

```text
抽象生成机制
类型公约的功能
结构压力
节奏规律
信息释放方法
阻力设计原则
文风变量
```

禁止迁移：

```text
具体角色设定
桥段顺序
专属能力
独特怪物
标志性世界观
组织名称和等级体系
具体副本机制
关系链组合
名场面顺序
高识别文风
```

## 4. 类型公约与高风险内容形状

不是所有相似都是风险。网文大量写法属于类型公约。

真正要警惕的是：通用公约被组合成了某本书的高识别内容形状。

| 层级 | 可用程度 | 例子 | 处理方式 |
| --- | --- | --- | --- |
| 类型公约 | 可用 | 觉醒、系统、官方组织、任务、副本、打脸、反杀、规则怪谈 | 可以用，但必须服务我们自己的主承诺和事件链 |
| 抽象机制 | 可迁移 | 先给异常现象，再给错误解释，主角测试规则并付出代价 | 可以进入 route/路线模型 |
| 内容形状 | 高风险 | 主角加入某特定机构后按相同事件顺序处理相似怪谈 | 只能观察，不能迁移 |
| 高识别元素 | 禁止迁移 | 专属能力表现、怪物视觉规则、标志性组织等级、名场面顺序 | 不进入运行包 |

## 5. 机制压缩标准

一条样本观察必须通过以下检查，才允许从“内容形状”压缩成“生成机制”。

| 检查项 | 问题 | 未通过表现 |
| --- | --- | --- |
| 去专名 | 是否去掉了人名、组织名、地名、怪物名、能力名后仍成立 | 去掉专名后就说不清 |
| 去顺序 | 是否不依赖原书同一事件顺序 | 仍然按原书 1-2-3 桥段推进 |
| 去视觉 | 是否没有沿用原书标志性视觉 | 怪物形态、能力画面、名场面近似 |
| 去关系 | 是否没有沿用原书角色关系组合 | 主角、搭档、上级、反派结构近似 |
| 去文风 | 是否没有沿用作者高识别语言皮肤 | 句式、口头禅、章末气口近似 |
| 多源支持 | 是否至少有 3 个样本或明确标为候选 | 只来自一本书 |
| 功能重述 | 是否能用“功能”而非“桥段”描述 | 仍在描述具体剧情 |

通过后的表达应该类似：

```text
可迁移机制：规则型异常事件常先给一个可观察异常，再给一个错误解释，随后让主角测试规则并付出代价。翻牌不能只解释规则，而要让原解法失效，逼出下一轮行动。
```

不合格表达：

```text
参考某书的镜子宿舍事件，让主角通过镜像延迟发现规则。
```

## 6. 提取证据的迁移边界标签

后续每条证据都必须标注 `transfer_boundary`。

| 标签 | 含义 | 是否能进入运行包 |
| --- | --- | --- |
| transferable_mechanism | 已压缩成抽象机制 | 可以 |
| genre_convention_rebuild | 类型公约，需要重构为我们自己的形状 | 可以，但必须改写 |
| observe_only | 只能观察，不迁移 | 不可以直接进入 |
| prohibited_shape | 高识别内容形状，禁止迁移 | 不可以 |
| legal_platform_review | 疑似硬风险，需合规复核 | 暂停 |
| soft_risk_review | 可能引发读者/作者圈相似感 | 暂停或强改写 |

## 7. 小说相似风险检查表

后续 clean-room 前 3 章、前 10 章计划和正文都要过这张表。

| 检查项 | 问题 | 风险等级 |
| --- | --- | --- |
| 角色相似 | 主角身份、性格、能力来源是否像某本书 | 高 |
| 桥段相似 | 前 3 章事件顺序是否像某本书 | 高 |
| 能力相似 | 能力表现、限制、升级方式是否过于贴近 | 高 |
| 组织相似 | 管理机构、等级体系、任务机制是否换皮 | 高 |
| 异常/怪物相似 | 核心视觉、行为规则、破解方式是否近似 | 高 |
| 副本机制相似 | 案件/副本的规则组合和反转路径是否近似 | 高 |
| 关系链相似 | 搭档、上级、反派、亲友牵连是否组合近似 | 中高 |
| 文风相似 | 是否像某作者语言皮肤 | 中高 |
| 爽点路径相似 | 打脸、反杀、翻牌是否沿用原路径 | 中高 |
| 标题简介相似 | 书名、简介、核心卖点是否让人联想到单一作品 | 中高 |

## 8. 对后续流程的约束

### P2 前 10 章结构证据抽取

每条证据新增：

```text
transfer_boundary
risk_note
```

没有迁移边界的证据，不能进入归纳。

### P2.5 路线候选判断

路线候选只能吸收：

```text
transferable_mechanism
genre_convention_rebuild
```

不能吸收：

```text
observe_only
prohibited_shape
legal_platform_review
soft_risk_review
```

### clean-room 前 3 章

生成前必须检查：

1. 是否只使用抽象机制。
2. 是否避开单一样本的桥段顺序。
3. 是否重构角色身份、能力表现、组织命名和异常规则。
4. 是否有软风险相似感。
5. 是否能解释每个核心元素来自“功能需求”，不是来自某个样本形状。

## 9. 当前临时红线

在没有更细平台协议和法律意见前，先执行以下临时红线：

1. 不复制、不改写、不近似改写任何样本文字。
2. 不复用单一样本的前 3 章事件顺序。
3. 不复用单一样本的能力表现、限制和升级路径组合。
4. 不复用单一样本的组织名称、等级体系和任务流程组合。
5. 不复用单一样本的异常/怪物视觉和破解规则组合。
6. 不复用单一样本的主角身份、搭档关系、上级关系和反派结构组合。
7. 不复用单一样本的名场面顺序。
8. 不复刻单一作者的高识别文风。
9. 任意机制若只来自一本书，只能标为 `route_candidate`，不能进入正式运行包。
10. 任意元素无法说明“功能必要性”，只因为样本里好看，禁止进入我们的项目。
