# 模块依赖与推进顺序

## 0. 当前结论

现在项目的问题不是“模块不够”，而是缺一个模块调度层。

如果没有调度层，每个问题都会显得很重要：

```text
平台规则要研究
收益要研究
AI / 抄袭风险要研究
编辑审稿要研究
大纲要研究
文风要研究
长期一致性要研究
基础叙事逻辑要研究
样本还要拆
clean-room 还要跑
```

这些都重要，但不能同时完整展开。当前必须按依赖顺序推进：

```text
先确定平台和风险边界
再校准前 10 章提取器
再判断 route_candidate
再深拆第一卷 / 事件链
再建立内容内核和运行包
再进入 clean-room 开局诊断
最后才激活完整长期一致性和正文生产系统
```

一句话：

```text
现在不是做所有模块，而是把每个模块放到正确的启动时机。
```

## 1. 批判性自问自答

### Q1：为什么用户会觉得乱？

因为当前路线图只定义了 P0-P8 的阶段顺序，但没有定义各模块在这些阶段里的状态。

比如“长期一致性”明显重要，但路线图没有说它现在是：

```text
马上做
只留接口
等 route 之后做
还是等正式连载前做
```

没有状态，模块就会互相抢优先级。

### Q2：为什么不能现在把长期一致性系统完整搭起来？

因为长期一致性系统依赖这些前置决策：

```text
主 route 是什么
主角靠什么持续行动
第一卷压力源是什么
能力体系如何制造剧情
关系线是否重要
文风和叙述距离是什么
平台更新节奏是什么
```

这些未确定前，提前做完整 `story_bible`、`style_bible`、`context_pack`，很容易把假设写成事实，后续反而难改。

### Q3：那长期一致性是不是可以先不管？

不能。

正确处理是：

```text
现在：记录长期一致性需求和风险信号。
P3/P4 后：开始定义我们项目需要哪些记忆字段。
P5/P6 前：启用最小 Story Bible / Style Bible。
正式连载前：启用完整 context_pack / project_state / human_decision_log。
```

也就是说，它不是现在完整生产，但必须现在进入调度板。

### Q4：编辑门、大纲、平台、合规是不是也一样？

不完全一样。

合规和相似风险是硬门禁，从 P0 开始一直生效。

平台机制现在必须补，因为它会影响 P2 字段。

编辑门现在只做审稿信号，不做完整投稿包。

大纲现在只做大纲学习接口，不写百万字大纲。

长期一致性现在只做风险观察，不建完整记忆系统。

### Q5：现在主线到底是什么？

当前主线只有一条：

```text
P0.6 三平台机制研究
-> P2 三本样本校准批次
-> P2.5 route_candidate 判断
```

其他模块如果不能服务这条线，就不能抢占执行资源。

## 2. 模块状态定义

以后任何新模块都必须先归类。

| 状态 | 含义 | 现在怎么处理 |
| --- | --- | --- |
| active_now | 当前主线必须做，不做会卡住下一步 | 立即推进 |
| always_on_gate | 全流程硬门禁，不是单独阶段 | 每次关键判断都检查 |
| interface_now | 现在只留字段 / 观察点，不完整展开 | 接入模板和审查项 |
| parked_until_trigger | 暂停完整建设，等触发条件 | 写清触发条件 |
| future_after_project_selection | 立项后才真正产品化 | 暂不消耗当前资源 |
| do_not_do_now | 现在做会污染判断或过早锁死 | 明确禁止 |

## 3. 当前模块状态表

| 模块 | 当前状态 | 为什么 | 触发 / 下一步 |
| --- | --- | --- | --- |
| P0.6 三平台机制和收益研究 | active_now | 平台收益、投稿路径、AI风险会影响 P2 字段和样本判断 | 先补到能支持 P2，不追求一次性穷尽 |

| 长期一致性 / 外部化故事记忆 | parked_until_trigger | 重要，但依赖 route、第一卷、人物和能力体系 | P5/P6 前启用最小 Story Bible；正式连载前启用完整记忆协议 |

## 4. 当前执行脊柱

当前只跑这条线：

```text
1. 补 P0.6 三平台机制研究
2. 把平台字段、编辑字段、长期风险字段接入 P2 使用
3. 选 3 本样本跑 P2 校准批次
4. 检查 P2 是否能产出 route_candidate
5. 如果不能，回修 P2 协议
6. 如果能，扩到 10-15 本前 10 章快拆
7. 进入 P2.5 路线候选判断
```

当前不跑：

```text
完整 Story Bible
完整 Style Bible
完整长篇记忆系统
完整投稿包
完整百万字大纲
完整作者 / 书籍 runtime skill
正式正文
```

## 5. 长期一致性模块的正确位置

长期一致性是长篇小说生产的核心问题，但它不是 P2 之前的主工作。

它分三层进入系统。

### 5.1 现在：只记录风险和需求

P2/P3 阶段只观察：

```text
样本是否有持续主线
能力 / 资源 / 关系是否会形成长期循环
前 10 章是否已经埋下后续承诺
读者追读是否依赖未解决问题
是否出现重复爽点和设定漂移风险
是否需要组织、任务、地图、敌人、伏笔表来维持一致性
```

这些是“长期一致性需求”，不是我们项目的正式设定。

### 5.2 P5/P6 前：启用最小 Story Bible / Style Bible

当 route_candidate 已经通过 P3/P4 验证，准备 clean-room 前 3 章或前 10 章计划时，必须启用最小版：

```text
主角核心欲望 / 外部压力
第一卷目标
能力体系边界
组织 / 敌人 / 规则最小表
主要关系状态
已埋承诺和待兑现清单
文风 / 叙述距离约束
禁止相似内容形状
```

这时不是写完整世界观，而是给开局诊断一个稳定底盘。

### 5.3 正式连载前：启用完整外部化故事记忆

一旦进入连续章节生产，必须从归档协议升级到 active schema：

```text
story_bible.yaml
style_bible.yaml
chapter_card.yaml
context_pack.yaml
project_state.json
human_decision_log.yaml
chapter_validation_report.yaml
```

执行规则：

```text
每章写前生成 context_pack
每章写后更新 project_state
每章记录人物状态、伏笔、资源、能力、关系和重复风险
人类关键决策写入 human_decision_log
```

参考归档：

```text
docs/90_前期规划归档/05_协议模板/长篇记忆管理协议.md
docs/90_前期规划归档/05_协议模板/story_bible模板.yaml
docs/90_前期规划归档/05_协议模板/style_bible模板.yaml
docs/90_前期规划归档/05_协议模板/context_pack模板.yaml
docs/90_前期规划归档/05_协议模板/project_state模板.json
```

## 6. 新模块进入系统的判断规则

以后出现任何新想法，先问四个问题。

### 问题 1：它会不会卡住当前 P0.6 -> P2 -> P2.5？

如果会，进入 `active_now`。

例子：

```text
平台 AI 政策
收益触发条件
P2 字段缺失
样本选择偏差
```

### 问题 2：它是不是硬风险？

如果是，进入 `always_on_gate`。

例子：

```text
抄袭 / 洗稿边界
平台违规
合同独家
AI 低质处罚
版权授权冲突
```

### 问题 3：它现在只需要观察，不需要完整建设吗？

如果是，进入 `interface_now`。

例子：

```text
大纲学习
文风信号
长期一致性需求
跨题材基础逻辑
改编潜力
```

### 问题 4：它是不是只有项目立项后才有意义？

如果是，进入 `parked_until_trigger` 或 `future_after_project_selection`。

例子：

```text
完整 Story Bible
完整 Style Bible
context_pack 自动化
正式章节生产流水线
投稿材料包
```

## 7. 接下来三步

### 第一步：补完 P0.6 到“够用”

不是把三平台研究做到百科全书，而是补到能回答：

```text
不同平台有哪些钱可以拿
触发条件是什么
直发 / 内投 / 签约路径差异是什么
AI 辅助和低质内容有什么风险
这些如何影响前 3 章、前 10 章、2 万字和 10 万字
```

### 第二步：跑 3 本 P2 校准批次

用 3 本样本测试 `chapter_evidence.yaml` 是否能稳定提取：

```text
读者承诺
前 10 章兑现
叙事因果
阻力反咬
翻牌 / 揭示
基础叙事逻辑候选
编辑审稿信号
大纲苗头
长期一致性风险
相似风险边界
```

如果提不出来，先修协议，不扩大样本。

### 第三步：做 P2.5 route_candidate 判断

判断这条 route 是否值得进入 P3 深拆：

```text
能不能留人
能不能写长
能不能 clean-room 迁移
能不能规避相似风险
能不能适配当前平台
能不能形成我们自己的 house style
```

## 8. 停止规则

出现以下情况，要暂停当前扩展：

```text
一个新模块开始抢主线，但说不清它服务哪个阶段
一个文档只增加概念，不改变下一步执行
一个结论无法回到样本或平台证据
一个方向需要完整世界观才能解释清楚
一个方案依赖“先相信它能写长”
一个 clean-room 开局像单本样本贴皮
一个 agent 把用户举例直接压缩成确定分类
```

停止后回到本文件，重新归类模块状态。

## 9. 当前总判断

长期一致性没有被否定，它被放到了正确位置。

编辑门、大纲、文风、基础逻辑、平台机制也都不是“以后再说”，而是按不同深度进入当前流程。

当前真正要避免的是：

```text
为了显得系统完整，把生产后期模块提前做成空架子。
```

更稳的顺序是：

```text
P0.6 够用
P2 校准
P2.5 判断
P3 深拆
P4 内核
P5 最小 Story Bible / Style Bible
P6 clean-room
P7 长篇记忆系统和正式生产准备
```

这才是从“很多事都要做”变成“知道先做什么、后做什么、什么时候不做”的控制板。
