# 从文案蒸馏迁移到小说

## 结论

小说项目不应该回到传统拆书法。传统拆书通常是人工列字段：人物、设定、冲突、节奏、爽点。这能帮助理解，但不能稳定复刻。

文案项目已经跑通的核心能力是：

```text
证据抽取
-> source-open 整篇通读归纳
-> 生成模型 / 语言渲染器拆分
-> route selector
-> runtime skill
-> clean-room 复刻
-> blind-review
-> 回到原文证据修 skill
```

小说这边要迁移这个方法，而不是重新发明一套古法拆书表。

## 文案侧对应关系

| 文案侧 | 小说侧迁移 |
| --- | --- |
| ASR / 整理稿 / 精修稿 | 样本来源、章节文本、简介标签、人工校对摘录 |
| API 结构证据抽取 | 小说结构证据抽取 |
| source-open 整篇通读 | 整书/卷/章节/场景多颗粒度通读 |
| 生成模型 | 单书生成模型 / 题材生成模型 |
| 语言渲染器 | 叙述渲染器 |
| route selector | 题材路线选择器 |
| runtime author skill | runtime book/genre skill |
| clean-room 复刻 | 新开局/第一卷 clean-room 生成 |
| blind-review | 追读感、番茄感、可连载性、改编潜力盲审 |

## 小说侧固定 runtime slots

这些 slot 不是传统表格字段，而是小说生成协议必须回答的运行参数。

1. Activation：当前激活的是哪类番茄赛道、哪种单书模型，禁止带入哪些自嗨倾向。
2. Dominant Law：这本书必须优先保护什么效果，例如追读、爽感、悬疑、压迫感、智斗、情绪补偿。
3. Reader Promise Engine：如何建立、兑现、升级、转移读者承诺。
4. Plot Causality Engine：为什么下一幕必须发生，前一个行动如何逼出后一个行动。
5. Opposition Engine：敌人、规则、误判、代价、组织压力如何反咬主角。
6. Reveal / Flip Engine：信息什么时候藏、什么时候放、什么时候误导、什么时候翻牌。
7. Serial Pacing Engine：每章如何制造追读，每 3-10 章如何兑现一次。
8. Support Function：设定、民俗、机构、地名、能力体系、案件素材分别用来支撑什么。
9. Adaptation Engine：哪些元素服务漫画、短剧、视觉化名场面。
10. Narrative Renderer：叙述距离、对白比例、动作密度、心理活动、句段呼吸。
11. Strength Knobs：爽点、悬疑、智斗、情绪、设定密度、现实感分别开到什么强度。
12. Fidelity Defects：这个赛道允许甚至需要哪些粗糙感，哪些“优化”会伤害番茄感。
13. Transfer Boundary：哪些机制可迁移，哪些只是某本书或某个作者的壳。
14. Drift Alarms：哪些信号说明我们写偏了，例如设定展览、慢热文学化、只比战力、案子不推进。
15. Validation Gate：如何判断生成稿是否能继续推进，而不是只像一个漂亮设定。

## 多颗粒度 source-open gate

短视频可以整篇通读。小说必须分颗粒度，否则成本会失控。

| 颗粒度 | 作用 | 最低读取 |
| --- | --- | --- |
| 整书层 | 判断商业承诺、长期赛道、主线和完结风险 | 核心样本才读完整 |
| 卷/副本层 | 判断事件链如何开始、升级、反转、收束 | 核心样本至少 1 个完整卷 |
| 前 30 章 | 判断番茄冷启动和追读 | 候选深拆样本必读 |
| 前 10 章 | 判断点击承诺是否兑现 | 所有样本必读 |
| 场景层 | 判断行动因果、对手反咬、翻牌执行 | 抽关键场景 |
| 段落层 | 判断叙述渲染器 | 抽高留存/高爽点/高反转段落 |

## 第一批提取器应该抽什么

不要先抽“人物设定表”。第一批应抽生成因果。

1. `reader_promise_evidence`：承诺如何建立、首次兑现在哪里、之后如何升级。
2. `plot_causality_evidence`：每个关键事件为什么发生，主角为什么不能不行动。
3. `opposition_evidence`：阻力来自谁，如何反咬，是否让主角付出代价。
4. `reveal_flip_evidence`：哪些信息被隐藏、误导、释放、翻牌。
5. `serial_pacing_evidence`：章末钩子、3 章兑现、10 章兑现、卷末钩子。
6. `adaptation_evidence`：视觉名场面、短剧钩子、漫画分镜、怪物/能力可视化。
7. `narrative_renderer_evidence`：对白、动作、心理、说明、转场的比例和气口。

## 为什么旧表不够

旧表会问：

```text
这本书的读者承诺是什么？
能力体系是什么？
翻牌方式是什么？
```

这会变成填空题，最后仍然靠人拍脑袋。

迁移后的方法要问：

```text
哪个证据证明它在第几章建立了承诺？
哪个情节动作逼出下一幕？
哪个对手动作改变了主角处境？
哪个隐藏信息让读者继续追？
这个机制在同 route 的几本书里是否重复出现？
按这个机制 clean-room 写一个新开局，盲审能不能过？
```

区别在这里：传统拆书得到“结论”，蒸馏流水线得到“可运行模型”。

## 当前小说项目的优先级

1. 先做番茄赛道模型，不急着做作者模型。
2. 先做单书层和 route selector，不急着模仿具体作者文风。
3. 先验证前 10 章和第一卷，不急着规划百万字全书。
4. 先验证可执行性和追读，不急着追求文学性。
5. 先建立 evidence -> skill -> clean-room -> review 闭环，再扩展自动化。
