# 小说专用生成部件流程

## 0. 总原则

文案侧的 `Progression Engine` 解决的是“一段话为什么推到下一段话”。小说侧不能照搬，因为小说的核心不是段落推进，而是连续行动、阻力、代价、信息差和追读。

所以小说侧保留文案蒸馏的流程骨架，但把运行部件改成：

```text
Reader Promise Engine
Plot Causality Engine
Opposition Engine
Reveal / Flip Engine
Serial Pacing Engine
Support Function
Adaptation Engine
Narrative Renderer
Validation Gate
```

每个部件都必须经历同一条证据链：

```text
样本证据抽取
-> source-open 多颗粒度通读
-> 横向归纳 route 规律
-> 写入 runtime skill
-> clean-room 生成
-> blind-review
-> 回到样本修正
```

任何部件都不能只靠“我觉得番茄小说应该这样写”来定规则。

## 1. Reader Promise Engine

### 解决什么问题

读者承诺解决的是：读者为什么点进来，又为什么愿意继续看。

它不是一句“这本书是爽文/悬疑/智斗”。长篇小说可以同时承诺爽点、悬疑、成长、关系、身份揭露和世界观探索，但必须区分主承诺、副承诺和阶段性承诺。

### 抽取流程

1. 看书名、简介、标签，记录点击前承诺。
2. 看第 1 章，记录开场实际承诺是否和简介一致。
3. 看前 3 章，记录承诺是否已经落到具体危机、能力、关系或事件。
4. 看前 10 章，记录第一次兑现发生在哪里。
5. 看前 30 章，记录承诺是否升级或转移。
6. 如果样本进入深拆，继续看一个完整卷/副本，记录卷级承诺如何收束。

### 证据字段

| 字段 | 含义 |
| --- | --- |
| pre_click_promise | 书名、简介、标签承诺了什么 |
| opening_promise | 第 1 章实际承诺了什么 |
| promise_carrier | 承诺由什么承载：身份、能力、危机、案件、关系、世界观 |
| first_payoff | 第一次兑现发生在第几章 |
| payoff_type | 打脸、破局、升级、反转、救人、揭露、情绪补偿 |
| promise_upgrade | 前 30 章是否从小承诺升级成大承诺 |
| promise_drift | 是否出现简介卖点和正文不一致 |

### 归纳输出

归纳时不要只写“这本书承诺爽点”。要写成运行规则：

```text
这个 route 的常见承诺建立方式是：
第 1 章先给主角处境压迫
第 2-3 章给异常/能力入口
第 5-10 章完成第一次小破局
第 10 章后把个人危机升级成组织/城市/规则危机
```

### 生成时怎么用

生成开局前必须先写：

```text
点击承诺：
开场承诺：
前 3 章承诺：
前 10 章兑现：
第一卷承诺升级：
```

如果这五项写不清，不能进入正文。

## 2. Plot Causality Engine

### 从文案 Progression Engine 改了什么

文案侧的 progression 是：

```text
误解
-> 反驳
-> 机制显形
-> 材料支撑
-> 判断收束
```

小说侧的叙事因果是：

```text
目标
-> 阻力
-> 行动
-> 后果
-> 新压力
-> 下一步被迫行动
```

核心区别是：文案可以靠论证推进，小说必须靠处境变化推进。小说里不能只是“然后发生了 B”，必须是“因为 A 导致 B，B 又逼出 C”。

### 抽取流程

1. 逐章记录主角当前目标。
2. 记录触发事件：为什么这一章必须开始。
3. 记录主角采取了什么行动。
4. 记录行动造成什么后果。
5. 记录这个后果如何改变主角处境。
6. 记录下一章是否由这个新处境自然逼出。

### 证据字段

| 字段 | 含义 |
| --- | --- |
| chapter_range | 章节范围 |
| protagonist_goal | 主角当前想要什么 |
| trigger | 什么事件逼他行动 |
| action | 主角做了什么 |
| consequence | 行动带来什么结果 |
| irreversible_change | 哪个处境不可逆地变化了 |
| next_pressure | 这个变化如何逼出下一章 |
| removable_scene_alarm | 如果删掉这一幕，后面是否仍然成立 |

### source-open 归纳重点

通读时要判断一条事件链是不是硬接：

```text
低质量：
主角去了学校，然后遇到怪物，然后打赢，然后又接到任务。

高质量：
主角为了隐藏能力去学校
但学校发生异常检测
他被迫出手救人
出手导致组织发现他
组织给他任务
任务又暴露更大的异常规则
```

### 生成时怎么用

每一幕生成前先写一行因果：

```text
因为【上一幕后果】，主角现在必须【行动】，但【阻力】会导致【新代价】。
```

如果这一行写不出来，这一幕大概率是硬塞。

### 失败警报

1. 删掉某章后，后续剧情仍然能成立。
2. 主角行动不是被处境逼出来，而是作者让他去。
3. 每章只换地点，不换压力。
4. 主角赢了但处境没有变化。
5. 事件结束后没有新问题、新代价或新钩子。

## 3. Opposition Engine

### 解决什么问题

对手/阻力系统解决的是：主角为什么不能一路平推。

这里的“对手”不是只有反派。阻力可以来自敌人、规则、组织、信息差、时间限制、身体代价、社会关系、舆论、制度、地理环境、民俗禁忌、异常本身。

### 阻力层级

| 层级 | 说明 | 都市高武/异常事件例子 |
| --- | --- | --- |
| 人物对手 | 有明确目标和行动的人 | 竞争者、队友、反派、上级、家族 |
| 组织阻力 | 规则、流程、保密制度、权限 | 国安局、749 类机构、学校、平台、官方封锁 |
| 异常规则 | 怪物、诅咒、禁区、能力反噬 | 不能直视、不能说真名、杀死会复制 |
| 信息差 | 主角不知道、读者不知道、对手知道 | 任务真相、身份隐藏、规则误导 |
| 资源限制 | 时间、人手、钱、装备、权限不够 | 24 小时内处理，不能调动武器 |
| 代价限制 | 使用能力会损失什么 | 寿命、记忆、污染、暴露身份 |
| 关系阻力 | 情感、道德、身份拖拽行动 | 亲人被牵连，队友不信任，救人与保密冲突 |
| 主角盲区 | 性格、经验、误判造成阻力 | 太自信、太善良、资料不足 |

### 抽取流程

1. 先记录主角目标。
2. 找出谁或什么在阻止目标。
3. 判断阻力是否主动行动，而不是被动等主角打。
4. 记录阻力手里有什么筹码。
5. 记录主角如何应对。
6. 记录应对之后付出什么代价。
7. 记录阻力是否升级或反咬。
8. 判断这个阻力能否在后续重复生产剧情。

### 证据字段

| 字段 | 含义 |
| --- | --- |
| protagonist_goal | 主角目标 |
| opposition_type | 阻力类型 |
| opposition_actor_or_rule | 具体对手或规则 |
| opposition_motive | 阻力为什么存在 |
| leverage | 阻力的筹码 |
| intervention | 阻力如何主动干预 |
| protagonist_response | 主角如何应对 |
| cost | 主角付出什么代价 |
| backlash | 阻力如何反咬 |
| escalation | 阻力是否升级 |
| repeatability | 这个阻力能否持续生产剧情 |

### 生成时怎么用

每个关键场景必须写清楚：

```text
主角想要：
阻力想要：
阻力筹码：
阻力动作：
主角反制：
主角代价：
下一轮反咬：
```

如果阻力没有动作，只是等着被主角解决，这个场景就会变成单向输出。

### 卷级阻力设计

一个完整事件卷里，阻力最好分三层：

```text
表层阻力：眼前怪物/案件/敌人
中层阻力：组织规则/信息误导/能力限制
底层阻力：世界规则/身份真相/更大势力
```

前 10 章可以只露出表层和一点中层。第一卷后半要让中层开始反咬。底层阻力用来接下一卷。

### 失败警报

1. 反派只会嘲讽，不会改变主角处境。
2. 怪物只是血条，不携带规则。
3. 组织只是背景板，不制造权限、保密、考核、误判。
4. 主角每次赢都无代价。
5. 阻力无法复用，打完就没了。

## 4. Reveal / Flip Engine

### 解决什么问题

揭示/翻牌系统解决的是：读者为什么在“知道主角大概率会赢”的情况下，仍然想看主角怎么赢。

番茄爽文不能只有强，也要有“暂时看不懂怎么赢”。翻牌不是为了炫反转，而是为了让信息释放带来爽点、悬疑或局势变化。

### 基本概念

| 概念 | 含义 |
| --- | --- |
| withholding | 暂时不告诉读者某个关键信息 |
| clue | 给读者可回收的线索 |
| misdirection | 让读者或角色形成错误解释 |
| reveal | 揭示真实信息 |
| flip | 揭示后局势反转 |
| payoff | 前面线索被兑现 |
| next mystery | 翻牌后打开新的问题 |

### 信息四层

每个关键秘密都要判断四类人知道什么：

```text
读者知道什么
主角知道什么
对手知道什么
组织/世界知道什么
```

追读感往往来自这四层的不一致。

### 抽取流程

1. 记录本章打开了什么问题。
2. 记录本章隐藏了什么信息。
3. 记录本章给了哪些线索。
4. 记录角色和读者是否被误导。
5. 记录什么时候揭示。
6. 记录揭示是否改变局势。
7. 记录翻牌后是否打开新问题。

### 证据字段

| 字段 | 含义 |
| --- | --- |
| open_question | 当前打开的问题 |
| hidden_information | 被隐藏的信息 |
| clue | 前置线索 |
| false_explanation | 临时错误解释 |
| reader_assumption | 读者可能以为什么 |
| reveal_point | 第几章揭示 |
| flip_effect | 揭示后局势如何变化 |
| payoff_target | 回收了哪个伏笔 |
| next_mystery | 翻牌后打开什么新问题 |

### 翻牌类型

| 类型 | 说明 |
| --- | --- |
| 能力翻牌 | 主角能力真实用法不是表面那样 |
| 身份翻牌 | 主角、对手、受害者、组织身份反转 |
| 规则翻牌 | 异常事件的真实规则被揭露 |
| 对手翻牌 | 真正敌人不是眼前的人 |
| 情绪翻牌 | 某个行动背后的情感动机反转 |
| 世界观翻牌 | 单个事件指向更大的世界规则 |

### 生成时怎么用

每 3-10 章要设计一次小翻牌或小兑现。第一卷要有一次大翻牌。

生成前写清楚：

```text
本阶段要打开的问题：
读者看到的表层解释：
真实解释：
前置线索：
误导信息：
翻牌章节：
翻牌后局势变化：
翻牌后新问题：
```

翻牌必须改变行动、关系、风险或目标。只让读者说“原来如此”，但剧情不变，是弱翻牌。

### 失败警报

1. 什么都藏，不给线索，最后靠作者宣布真相。
2. 翻牌只解释过去，不推动未来。
3. 反转过密，读者失去稳定承诺。
4. 太早解释完规则，后续只剩打怪。
5. 主角早就知道一切，读者被单方面蒙在鼓里，容易产生欺骗感。

## 5. Serial Pacing Engine

### 解决什么问题

连载节奏解决的是：每章是否有继续读下一章的理由。

小说不是一次性看完的文章。番茄尤其要关注前 3 章、前 10 章、前 30 章和卷末节点。

### 抽取流程

1. 逐章记录本章目标。
2. 记录本章新增冲突。
3. 记录本章兑现了什么。
4. 记录本章留下什么未解决问题。
5. 记录章末钩子类型。
6. 每 3 章检查是否有小变化。
7. 每 10 章检查是否有小兑现。

### 节奏单位

```text
1章：一个明确推进
3章：一个小压力升级
10章：一个承诺兑现
30章：一个较完整事件链
1卷：一个完整副本/事件闭环
```

### 章末钩子类型

| 类型 | 说明 |
| --- | --- |
| 危机钩 | 危险出现或升级 |
| 信息钩 | 新秘密出现 |
| 行动钩 | 主角即将做一个高风险动作 |
| 关系钩 | 关系破裂、误会、身份暴露 |
| 奖励钩 | 新能力、新资源、新权限 |
| 反咬钩 | 主角刚赢，代价马上出现 |

### 失败警报

1. 章节只完成信息说明，没有行动变化。
2. 连续多章只铺设定，不兑现。
3. 章末没有未完成压力。
4. 每章都危机，但没有节奏层次。
5. 前 10 章没有让读者确认这本书的核心爽感模式。

## 6. Support Function

### 从文案支撑物改了什么

文案里的支撑物是历史材料、数据、案例、现代比喻。小说里的支撑物是设定、地名、民俗、组织、能力体系、案件素材、职业细节、规则。

它们不能只是好玩素材，必须服务某个运行功能。

### 功能分类

| 支撑物 | 应该服务什么 |
| --- | --- |
| 能力体系 | 制造行动可能性和限制 |
| 民俗/地域 | 制造陌生感、禁忌、规则 |
| 官方组织 | 制造权限、保密、任务、考核 |
| 案件素材 | 制造问题、线索、误导、翻牌 |
| 城市空间 | 制造现实感、追捕、封锁、目击 |
| 历史传说 | 制造底层规则或世界观钩子 |
| 装备道具 | 制造解法、代价、限制 |

### 失败警报

1. 设定解释很多，但不改变主角行动。
2. 民俗只是装饰，没有规则和禁忌。
3. 组织只发任务，不制造限制。
4. 能力只负责变强，不制造代价。
5. 地名只是旅游清单，没有事件功能。

## 7. Adaptation Engine

### 解决什么问题

改编潜力不是后期再考虑。立项时就要判断这个方向是否容易漫画化、短剧化、视觉化。

### 抽取流程

1. 标注强视觉场面。
2. 标注角色一眼能懂的人设标签。
3. 标注怪物、异常、能力是否可视化。
4. 标注每集短剧是否能有单集钩子。
5. 标注场景是否能低成本拍摄或生成。
6. 判断名场面是否能被切成短视频传播。

### 证据字段

| 字段 | 含义 |
| --- | --- |
| visual_scene | 强视觉场景 |
| character_tag | 角色标签 |
| anomaly_visualization | 异常/怪物/能力如何可视化 |
| episode_hook | 单集钩子 |
| production_difficulty | 拍摄/生成难度 |
| shareable_moment | 是否有传播名场面 |

### 失败警报

1. 爽点都在内心独白里，难视觉化。
2. 能力表现抽象，漫画和短剧不好呈现。
3. 场景过大但没有明确镜头中心。
4. 角色标签不清，改编后无法一眼识别。

## 8. Narrative Renderer

### 从语言渲染器改了什么

文案侧语言渲染器关注口播节奏、句长、判断句、比喻、金句。小说侧叙述渲染器关注的是阅读阻力和沉浸感。

### 抽取流程

1. 抽高爽点段落，看动作、对白、心理、说明比例。
2. 抽高悬疑段落，看信息隐藏和感官描写。
3. 抽转场段落，看作者如何从一个场景切到下一个场景。
4. 抽章末段落，看钩子如何落下。
5. 标注叙述距离：贴近主角、旁观解释、全知说明。

### 渲染字段

| 字段 | 含义 |
| --- | --- |
| pov_distance | 叙述距离 |
| dialogue_ratio | 对白比例 |
| action_density | 动作密度 |
| inner_monologue_density | 内心戏密度 |
| exposition_style | 设定说明方式 |
| transition_breath | 转场气口 |
| ending_breath | 章末收束方式 |

### 失败警报

1. 设定说明压过行动。
2. 对白只传递信息，没有冲突。
3. 心理活动解释太多，削弱节奏。
4. 章末太圆满，没有下一章压力。
5. 语言变得过度文学化，失去番茄阅读速度。

## 9. Validation Gate

### 盲审问题

小说侧不能只问“像不像”。必须问：

1. 前 3 章是否想继续看。
2. 前 10 章是否兑现过一次承诺。
3. 主角是不是被处境逼着行动。
4. 对手或规则是否会反咬。
5. 是否有信息差和翻牌。
6. 是否能继续生产 30 章以上。
7. 是否有番茄感，而不是设定自嗨。
8. 是否有漫画/短剧改编潜力。

### 评分建议

| 维度 | 低分表现 | 高分表现 |
| --- | --- | --- |
| 追读 | 看完一章没有下一章压力 | 每章都有未完成问题 |
| 因果 | 情节靠作者安排 | 前一幕后果逼出下一幕 |
| 阻力 | 主角单向输出 | 对手/规则持续反咬 |
| 翻牌 | 随机反转或无反转 | 有线索、有误导、有兑现 |
| 兑现 | 一直铺设定 | 前 10 章完成小破局 |
| 可连载 | 只有一个创意 | 能持续生产事件链 |
| 可改编 | 全靠内心和概念 | 有强视觉场景和角色标签 |

## 10. 最终工作流

第一轮样本阶段：

```text
收集 20-30 本同赛道样本
-> 每本记录书名/简介/标签/榜单/采样日期
-> 所有样本读前 10 章
-> 筛出 5-8 本深拆前 30 章
-> 筛出 2-3 本读完整卷/副本
```

第二轮抽取阶段：

```text
reader_promise_evidence
plot_causality_evidence
opposition_evidence
reveal_flip_evidence
serial_pacing_evidence
support_function_evidence
adaptation_evidence
narrative_renderer_evidence
```

第三轮归纳阶段：

```text
按单书写 source-open 备忘
-> 按 route 横向归纳
-> 区分稳定机制、题材壳、作者个人习惯
-> 写成 runtime book/genre skill
```

第四轮生成阶段：

```text
选择 route
-> 激活 runtime skill
-> 写点击承诺和前 10 章兑现计划
-> 写第一卷事件链
-> 写前 3 章 clean-room 开局
```

第五轮验证阶段：

```text
blind-review
-> 标出追读断点、因果断点、阻力断点、翻牌断点
-> 回到样本证据确认
-> 修改 runtime skill
-> 再生成
```

这个流程的目标不是“拆懂一本书”，而是得到可运行、可验证、可反复生成的番茄小说模型。
