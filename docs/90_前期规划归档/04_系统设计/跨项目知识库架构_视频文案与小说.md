# 跨项目知识库架构：视频文案与小说

更新时间：2026-05-04

## 结论

视频文案和小说可以共用一套知识库方法，但不能混用同一套执行记忆。

推荐架构是：

```text
共享知识层：通用概念、平台规则、结构 skill、文笔 skill、风险规则
↓
项目工作区：视频文案项目、小说项目分别独立
↓
项目主账本：各自的 state、style、memory、output
```

一句话：

`共享方法和知识，不共享项目状态。`

## 为什么可以共用

视频文案和小说有共同部分：

- 钩子。
- 情绪曲线。
- 冲突设计。
- 反转。
- 爽点。
- 追读/完播。
- 人设吸引力。
- AI味控制。
- 平台风控。
- 样本观察和 skill 蒸馏方法。

这些可以进入共享知识层，形成长期复利。

## 为什么不能混用

视频文案和小说的执行目标不同：

- 视频文案追求短时间抓注意、完播、转粉、转化。
- 小说追求长线追读、人物连续性、伏笔回收、章节节奏。

如果混用执行记忆，会出现污染：

- 小说章节被写成短视频口播腔。
- 视频文案被写成小说叙事腔。
- 视频爆款套路误导小说节奏。
- 小说人物状态污染文案项目。
- 不同平台风险规则混在一起。

## 分层规则

共享层可以放：

- 通用写作概念。
- 平台政策摘要。
- 公开样本观察笔记。
- 结构 skill。
- 文笔 skill。
- anti-AI 规则。
- 风控规则。

视频项目独立保存：

- 抖音账号/作者 baseline。
- ASR transcript。
- 视频脚本。
- 完播/互动数据。
- 文案结构复盘。
- 账号风格配置。

小说项目独立保存：

- story bible。
- style bible。
- project state。
- context pack。
- 章节策划卡。
- 人物状态。
- 伏笔索引。
- 章节全文和摘要。

## Skill 复用规则

可以跨项目复用的 skill：

- 开头钩子。
- 反转机制。
- 情绪递进。
- 用动作替代情绪直说。
- 禁用 AI 腔表达。
- 信息差制造期待。

需要改写后才能复用的 skill：

- 短视频“三秒钩子”用于小说时，要变成“章首钩子”或“前三章钩子”。
- 视频完播结构用于小说时，要变成“章节追读结构”。
- 口播节奏用于小说时，要弱化口语标签，转成叙事节奏。

禁止跨项目复用的内容：

- 具体视频文案原句。
- 具体小说桥段。
- 具体作者风格。
- 具体账号人设。
- 具体小说人物和世界观。

## 检索策略

写小说时：

- 优先检索小说项目本地索引。
- 其次检索共享 structure/prose/risk skill。
- 不默认检索视频 transcript 原文，除非明确做题材观察。

写视频文案时：

- 优先检索视频项目素材和账号 baseline。
- 其次检索共享钩子、情绪曲线、anti-AI 规则。
- 不检索小说正文，避免叙事腔污染。

## 最终判断

知识库方法可以统一，项目记忆必须隔离。

最合理的设计是：

`一个共享知识库 + 多个独立项目工作区 + 各自本地主账本 + 有边界的 skill 复用。`
