# 小说蒸馏迁移思考

更新时间：2026-05-09

## 0. 这份文档回答什么

这份文档回答的是：

- 当前文案蒸馏流水线成熟到了哪一步。
- 这套结构能不能迁到小说。
- 如果能迁，哪些层可以直接沿用，哪些层必须重构。
- 小说蒸馏不能只按“作者风格”理解，还要按什么颗粒度拆。

一句话判断：

> 当前文案蒸馏已经足够支撑“小说蒸馏 1.0”的搭建，但前提是不要把小说理解成“更长的文案”，而要改成“多颗粒度叙事系统蒸馏”。

## 1. 对当前文案蒸馏的判断

当前成熟度已经明显超过“prompt 仿写”。

现有能力不是简单的：

- 学几个句式
- 加几个反向限制
- 让模型别太 AI

而是已经形成了比较完整的 source model pipeline：

`精修原文 -> API 证据抽取 -> source-open 通读归纳 -> generation model / language renderer 拆分 -> route selector -> runtime skill -> clean-room 复刻 -> blind review -> 回原文证据修 skill`

这套体系当前最重要的成熟点有四个：

### 1.1 source-open gate 已经成立

现在不是只看抽取出来的证据点，而是要求：

- 必须回到精修全文
- 必须看整篇运动
- 必须区分“证据点支持”和“整篇力场支持”

这一步非常关键。

因为小说更怕“证据点过拟合”，没有这一层，后面一定会失真。

### 1.2 generation model / language renderer 已经拆开

这意味着你们已经知道：

- 结构像，不代表作者像
- 语言像，也不代表写作引擎像

对应到小说，这一步仍然成立，而且更重要。

### 1.3 route selector 已经被一代书生验证有效

一代书生当前最值钱的成果，不是复刻出一篇像稿，而是证明了：

> 同一作者内部确实存在 topic route 变体，而不是所有题都套一个母模板。

这对小说是硬需求。

因为小说作者的“题材 route”“叙事 route”“人物 route”会比文案更强。

### 1.4 blind review 已经不再只是“像不像”

虽然现在还是以“像不像作者”为主，但流程上已经形成了：

- 先 clean-room
- 再盲审
- 再回原文修 skill

这说明你们已经有了“模型闭环”，而不是一轮性产物。

## 2. 哪些东西可以直接迁到小说

下面这些层，小说可以直接沿用：

### 2.1 证据纪律

- 原文优先
- 精修版优先
- 抽取证据和人工通读分开
- blind review 不能直接变成源规则

这些都应该保留。

### 2.2 clean-room 纪律

小说更需要 clean-room。

因为小说更容易被：

- 热门模板
- 执行者自己的趣味
- 平台套路幻觉
- 其他作者残留

污染。

### 2.3 generation / renderer 拆分

小说同样需要：

- 生成模型：冲突、推进、结构、关系、承诺
- 语言渲染器：句子、叙述距离、对话感、章尾气口、心理描写、转场习惯

### 2.4 route selector

小说版一定要保留 route selector。

否则最容易犯两个错：

- 把同一作者不同题材变体误判成不同作者
- 把同一作者所有书硬压成一个母模板，写得非常假

## 3. 哪些东西不能直接照搬

当前文案蒸馏最强的部分，是“作者怎么理解一个题并把它讲透”。

小说不够。

因为小说不只是“讲什么”，还包括：

- 谁为什么必须行动
- 什么阻止他
- 读者为什么继续追
- 信息什么时候给，什么时候藏

所以文案模型不能原封不动迁到小说。

必须从“观点推进模型”升级成“叙事推进模型”。

## 4. 小说不能只用一个颗粒度学

这是迁移时最需要改的地方。

文案通常一篇就是一个完整运动单位。

小说不是。

小说至少要分四个稳定颗粒度：

### 4.1 书级

回答：

- 这本书总承诺什么。
- 这本书长期靠什么活着。
- 为什么它能写到几十万/上百万字不散。

### 4.2 卷 / 单元级

回答：

- 这一卷负责兑现哪部分承诺。
- 这一卷主冲突是什么。
- 这一卷如何喂养主线。

### 4.3 章节级

回答：

- 这一章靠什么让人点下一章。
- 这一章释放什么信息。
- 这一章留下什么债。

### 4.4 场景级

回答：

- 谁要什么。
- 为什么拿不到。
- 哪个动作让局面更糟、更深或更近。
- 这一场为什么逼出下一场。

一句话：

> 小说不能只按一本书学，也不能只按一句风格学，必须按“书级 / 卷级 / 章节级 / 场景级”一起学。

## 5. “读者承诺模型”不是单选题

小说的承诺不是：

- 爽点
- 悬疑
- 翻牌
- 关系拉扯
- 成长
- 压迫感
- 智斗

七选一。

更准确的理解是：

### 5.1 主承诺

这本书最核心卖给读者的满足。

### 5.2 副承诺

辅助维持追读、增强层次感的满足。

### 5.3 承诺会按颗粒度切换

比如：

- 书级主承诺：异常事件破局
- 书级副承诺：成长升级 / 阴谋追踪
- 卷级承诺：悬疑 + 规则破解
- 某几章承诺：压迫感 + 反杀快感
- 某条关系线承诺：信任建立 / 队伍磨合

所以承诺模型不是“一本书只能吃一种东西”。

而是：

> 一本书有主承诺，多个副承诺；不同颗粒度上，承诺的功能也不同。

## 6. 小说不是固定部件清单，而是推进系统

前面提到的：

- 对手 / 阻力系统
- 揭示 / 翻牌系统

不是说每一部小说都必须一模一样地具备这些表型。

更准确的说法应该是：

### 6.1 每部小说都必须有推进力

但推进力来源不同。

可能来自：

- 对手压制
- 生存环境
- 规则限制
- 关系误解
- 身份落差
- 情绪负债
- 智斗破局
- 升级回报

### 6.2 因此小说蒸馏不该先问“有没有某模块”

而该先问：

- 这本书主要靠什么让人往下看。
- 这种驱动力在书级、卷级、章节级分别怎么实现。

也就是说，小说版不应是“功能件 checklist”，而应是“推进问题框架”。

## 7. 当前赛道下，哪些推进系统最重要

如果目标是当前讨论的：

- 番茄
- 都市高武
- 都市异常事件
- 机构处置 / 749 / 国安局 / 收容

那么最重要的推进系统，确实更偏下面这些：

- 规则识别
- 破绽寻找
- 信息差
- 局势加压
- 假解与误导
- 翻牌破局

也就是更接近：

- 老千式推进
- 神秘复苏式拉扯

这不是说所有小说都这样。

而是说：

> 你当前要做的这个赛道，最吃这种“主角能赢，但读者暂时不知道怎么赢”的推进快感。

## 8. 小说版建议分层

如果把小说蒸馏单独建模，我建议先固定成 6 层。

### 8.1 作者层

回答：

- 作者稳定相信什么。
- 作者偏爱什么冲突。
- 作者的人物通常怎么思考、怎么失控、怎么赢。

### 8.2 书级承诺层

回答：

- 这本书整体卖什么。
- 主承诺是什么。
- 副承诺是什么。

### 8.3 卷 / 单元层

回答：

- 每一卷负责兑现哪部分承诺。
- 每一卷如何向主线输血。

### 8.4 章节推进层

回答：

- 每章靠什么推进。
- 章尾靠什么留读者。

### 8.5 场景因果层

回答：

- 谁要什么。
- 什么在阻碍。
- 场景如何制造新问题。

### 8.6 语言渲染层

回答：

- 句长
- 叙述距离
- 心理戏密度
- 对话节奏
- 转场习惯
- 收尾气口

## 9. 小说版 runtime slots 建议

基于现有文案模型，可以改成小说版 slots：

- Activation
- Platform Fit
- Author Route
- Book Route
- Promise Model
- Premise Transformer
- Character Operating Model
- Opposition Model
- Progression Engine
- Reveal Engine
- Worldbuilding Function
- Language Renderer
- Fidelity Defects
- Drift Alarms
- Validation Gate

这里最关键的新槽位有：

### 9.1 Promise Model

这本书主承诺和副承诺分别是什么。

### 9.2 Character Operating Model

主角通常如何判断、试探、冒险、退让、翻牌。

### 9.3 Opposition Model

阻力来自哪里：

- 人
- 规则
- 环境
- 信息
- 情绪
- 身份

### 9.4 Reveal Engine

什么时候藏信息，什么时候放信息，什么时候让读者误判，什么时候翻牌。

## 10. 验证方式也要升级

文案可以主要问：

- 像不像这个作者。

小说不够。

小说版 blind review 最少要多加四个问题：

- 像不像这个作者。
- 愿不愿意继续看下一章。
- 从哪里开始掉。
- 翻牌是否有前铺，还是临时编出来的。

因为小说的失败很多不是“不像作者”，而是：

- 像，但不追
- 像，但中段塌
- 像，但翻牌假
- 像，但人物是空心的

## 11. 当前最合理的迁移策略

不建议一步到位做“小说全模型”。

建议分三步：

### 第一步：保留现有顶层流水线

继续保留：

- 精修
- 抽取
- source-open
- generation / renderer
- clean-room
- blind review

### 第二步：先把非虚构 slots 改成小说版四问框架

先不急着写死一整套大 schema。

先按四层问题框架稳定下来：

- 书级四问
- 卷级四问
- 章节级四问
- 场景级四问

### 第三步：再从四问稳定出小说版 runtime slots

这样更稳，不容易从第一天就把 schema 写死。

## 12. 现阶段最值得优先研究什么

如果后续立刻开始小说蒸馏，我认为最优先的不是“文笔”，而是：

### 12.1 读者承诺层

先搞清一本书到底在卖什么。

### 12.2 场景因果层

先搞清场景为什么逼出下一场。

### 12.3 章尾续读机制

先搞清作者靠什么把人留到下一章。

### 12.4 翻牌 / 破局机制

尤其是当前目标赛道：

- 老千
- 神秘复苏
- 异常事件流

这一步会决定追读。

## 13. 当前结论

如果把这一轮思考压成一句话：

> 文案蒸馏已经成功证明“作者模型 + 语言渲染器 + route selector”是可行的；小说蒸馏下一步要做的，不是复制一套更长的文案框架，而是在这个基础上加上“多颗粒度承诺、叙事因果和续读验证”。

后续如果继续推进，建议下一份文档直接写：

- 小说蒸馏四层问题框架
- 番茄长篇的书级 / 卷级 / 章节级 / 场景级证据抽取模板
