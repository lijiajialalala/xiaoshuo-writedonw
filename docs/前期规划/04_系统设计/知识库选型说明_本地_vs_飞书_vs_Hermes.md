# 知识库选型说明：本地系统 vs LLM Wiki vs Hermes

更新时间：2026-05-04

## 结论

最终版不应该绑定 Hermes，也不应该把任何单一文档平台当成唯一知识库。

最稳架构是：

```text
本地文件 + SQLite 本地知识库 = 系统主账本和执行记忆
LLM Wiki 方法 = 知识沉淀和组织规范
Codex = 当前执行入口
Hermes = 未来可选的长期代理/调度器
```

## 四者分工

| 组件 | 定位 | 是否必须 | 适合做什么 |
|---|---|---:|---|
| 本地文件 | 主账本 | 必须 | story bible、style bible、project state、人类决策、伏笔状态 |
| SQLite/本地知识库 | 检索层 | 推荐 | 章节全文检索、人物事件、伏笔事件、审稿记录、规则资料 |
| LLM Wiki | 知识沉淀层 | 推荐 | source、concept、entity、skill、index、log 等长期知识制品 |
| Hermes | 长期代理层 | 可选 | 跨会话记忆、自动调度、多工具调用、长期守护 |

## LLM Wiki 怎么用

这里说的不是“飞书当知识库”，而是参考该文档里的 LLM Wiki 方法论：知识库像代码库一样管理，知识在摄入时就被整理成持久化、可维护、可迭代的制品。

LLM Wiki 适合沉淀：

- source：平台规则、调研材料、公开样本观察。
- concept：爽点、追读、AI味、洗稿风险、长篇记忆等概念。
- entity：番茄、七猫、具体小说项目、具体角色、具体平台。
- skill：结构 skill、文笔 skill、anti-skill。
- index：知识目录。
- log：知识库操作日志。

LLM Wiki 不应该替代执行主账本：

- `story_bible.yaml` 仍然是故事事实主账本。
- `style_bible.yaml` 仍然是文风事实主账本。
- `project_state.json` 仍然是章节状态主账本。
- SQLite 仍然负责机器检索和本地索引。

也就是说，LLM Wiki 是知识沉淀层，本地主账本是创作执行层。

## Hermes 怎么用

Hermes 如果以后接入，适合做“总管”：

- 记住你的长期偏好。
- 知道每个小说项目在哪里。
- 调度脚本生成 context pack。
- 调用本地知识库检索旧章节。
- 触发审稿、更新日志、生成复盘。

但 Hermes 不应该成为小说记忆唯一存储。它应该调用本地系统，而不是替代本地系统。

## 当前推荐路线

第一阶段只用：

- Codex。
- 本地 Markdown/YAML/JSON。
- SQLite + FTS5。

第二阶段再考虑：

- 向量库，用于语义相似和桥段相似。
- 飞书同步，用于项目看板和人工阅读。

第三阶段再考虑：

- Hermes，作为长期代理和多项目调度器。

## 最终判断

LLM Wiki 可以作为“知识组织方法”，Hermes 可以作为“代理入口”，但小说系统的核心必须是“本地可执行知识库”。

也就是说：

```text
你可以通过 Codex 使用这个系统。
以后也可以通过 Hermes 使用这个系统。
LLM Wiki 可以沉淀这个系统的知识制品。
但系统本体不属于 Codex、Hermes 或任何文档平台。
```
