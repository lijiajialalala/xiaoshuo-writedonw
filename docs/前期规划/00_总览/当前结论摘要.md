# 当前结论摘要

更新时间：2026-05-04

## 一句话结论

AI写小说商业上可试，但不能把核心策略建立在“绕过AI识别”上。更可行的路线是：用AI做爆款规律分析、章节策划、草稿生成和风控检查，由人类负责创作方向、方案取舍、章节讨论、最终验收和证据链保存。

当前最佳流程是：

`人类主创 + AI数据策划 + AI执行写作 + 本地主账本 + 本地知识库检索 + 长篇记忆包 + 三层审稿 + 人类验收`

## 当前平台选择

建议先围绕番茄小说和七猫中文网做“双平台双项目”，不要全网乱投。

番茄适合做自然流量测试：

- 优势：流量池大，算法分发强，平台自身公开提供AI辅助工具。
- 风险：不给量、首秀跳水、验证期失败、停推、冻结未发稿酬等风险更偏算法和内容质量。
- 适合：大众爽文、节奏强、可快速获得数据反馈的项目。

七猫适合做签约路径测试：

- 优势：签约/保底/编辑审稿路径更明确。
- 风险：合同追责更硬，公开条款中有停止支付稿酬、退回稿酬、解约、追责等工具。
- 适合：开篇完成度高、大纲清楚、人类主创痕迹强的项目。

## 关于AI使用

我们讨论后修正了“AI角色”的定义。

不推荐的模式：

- AI直接决定题材、大纲、人设、节奏、正文和修改。
- 人只做简单确认或上传。
- 目标是“去AI味”和“躲检测”。

推荐的模式：

- AI先基于爆款样本提出章节目的、情绪曲线、爽点位置、风险点。
- 人类和AI讨论方案，决定保留、删除、调整。
- AI根据人类确认后的章节策划卡生成草稿。
- 人类验收方向，再由AI执行具体修改。
- 全过程保存版本、讨论记录和采纳理由。

## 关键风险

平台风险主要不是“用了AI”四个字本身，而是以下几类：

- 被认定为低质AI生成内容。
- 被认定为非本人自主、独立创作。
- 被认定为AI洗稿、旧稿挪用、自我抄袭或侵权。
- 被认定为工作室投稿、多账号批量产文。
- 爆更、风格突变、桥段重复导致算法或编辑怀疑。

## 当前操作原则

- 不碰抄袭，不拿别人的书喂AI改写。
- 不做同一本书多平台投稿。
- 不做换名旧稿、套设定、套大纲。
- 不在七猫做纯AI代写测试。
- 番茄可以做更强的数据化试错，七猫要保守。
- 每本书都要独立设定、独立大纲、独立人设、独立证据链。

## 新增关键判断：长篇记忆

AI写长篇时，后期忘前文是高概率问题，不能只靠聊天上下文解决。

当前系统必须外部化记忆：

- `story_bible.yaml`：保存人物、设定、世界规则、伏笔和阶段目标。
- `style_bible.yaml`：保存文风、禁用AI腔、对话规则和偏好表达。
- `project_state.json`：保存章节状态、字数、确认状态、审稿状态和发布状态。
- `context_pack.yaml`：每章写前生成的最小必要记忆包。
- `human_decision_log.yaml`：保存人类创作判断和修改理由。
- 本地知识库：保存章节全文、摘要、人物事件、伏笔事件、规则资料和 skill，负责检索辅助。

每章写前读取 `context_pack`，每章写后更新人物状态、伏笔、章节摘要和风险报告。否则小说越长，人物、设定和文风越容易漂移。
