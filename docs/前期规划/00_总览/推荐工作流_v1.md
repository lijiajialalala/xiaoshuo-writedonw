# AI辅助小说创作推荐工作流 v1

更新时间：2026-05-04

## 核心定位

当前最佳方案不是“AI代写小说”，也不是“纯靠超长上下文写完整本书”，而是：

`人类主创 + AI数据策划 + AI执行写作 + 本地主账本 + 本地知识库检索 + 长篇记忆包 + 三层审稿 + 人类验收`

主原则：

- 人类负责方向、取舍、审美、确认和发布责任。
- AI负责分析、拆解、备选、草稿、修改和检查。
- 本地文件是官方记忆，模型上下文只是临时工作区。
- 本地知识库负责检索旧章节和规则，不替代结构化主账本。

## 推荐总流程

```text
平台风控与项目立项
↓
公开样本观察与结构 skill 提炼
↓
建立 Story Bible + Style Bible
↓
建立 project_state + 本地知识库
↓
生成当前章 context_pack
↓
AI 输出章节策划卡
↓
人类讨论确认
↓
AI 生成正文草稿
↓
结构审稿 + 文笔审稿 + 记忆审稿 + 平台风险审稿
↓
人类验收或要求修改
↓
写后更新主账本和知识库
↓
发布/暂存/复盘
```

## 主账本与知识库分工

主账本负责“事实正确”：

- `story_bible.yaml`：人物、设定、世界规则、伏笔、阶段目标。
- `style_bible.yaml`：文风、句式、禁用AI腔、对话规则。
- `project_state.json`：章节状态、确认状态、审稿状态、发布状态。
- `human_decision_log.yaml`：人类确认、否决和修改理由。
- `foreshadowing_index.yaml`：伏笔和回收状态。

本地知识库负责“查找相关内容”：

- 已写章节全文。
- 每章摘要。
- 人物出场记录。
- 旧决策记录。
- 平台规则。
- 结构 skill 和文笔 skill。
- 公开样本观察笔记。

写作时以主账本为准；知识库检索结果只能作为辅助证据，不能覆盖主账本。

## 每章标准流程

1. 写前检索

从本地知识库查找当前章相关信息：

- 最近3章发生了什么。
- 本章出场人物上次出现在哪。
- 当前人物知道什么、不知道什么。
- 未回收伏笔有哪些。
- 最近重复过哪些爽点、动作和结尾钩子。
- 目标平台本章可能有哪些风险。

2. 生成 context_pack

把检索结果和主账本压缩成当前章最小记忆包。只放本章真正需要的内容。

3. 生成章节策划卡

AI输出本章目的、情绪曲线、冲突来源、爽点位置、结尾钩子、不能做什么、需要人类确认的问题。

4. 人类讨论确认

人类确认方向，提出修改或否决。未确认前不进入正文生成。

5. AI生成正文草稿

AI根据 `context_pack`、章节策划卡、Story Bible 和 Style Bible 写草稿。

6. 四层审稿

- 结构审稿：本章目标、冲突、爽点、追读钩子是否成立。
- 文笔审稿：是否有AI腔、句式漂移、对话解释、抽象情绪。
- 记忆审稿：是否违背前文、人设、伏笔、世界规则。
- 风险审稿：是否有低质AI、洗稿、平台敏感、工作室化信号。

7. 人类验收

人类判断是否达到本章目的。具体修改可以继续让AI执行，但修改方向由人类确认。

8. 写后更新

更新：

- 章节摘要。
- 人物状态。
- 伏笔状态。
- 重复动作和爽点记录。
- 风险报告。
- 人类决策日志。
- project_state。
- 本地知识库索引。

## 技术实现建议

第一阶段：

- Markdown/YAML/JSON 作为主账本。
- SQLite + FTS5 做本地全文检索。
- 暂时不强依赖向量库。

第二阶段：

- 加入向量检索，用于语义相似桥段检测、旧章节相关片段召回。
- 可选 Chroma、LanceDB、Qdrant 或 FAISS。

第三阶段：

- 接入 Hermes 或其他 agent memory 作为调度器。
- Hermes 记住项目流程和用户偏好，但不替代本地主账本。

## 当前最重要的判断

长篇小说系统最怕三件事：

- 写到后面忘前文。
- 文风越来越像默认AI。
- 结构越来越模板化和重复。

所以当前最佳流程的核心不是“一次写更多字”，而是：

`每章写前准确加载记忆，每章写后稳定更新记忆，每章都有人类确认创作方向。`
