# 小说结构蒸馏协议

更新时间：2026-05-04

## 目的

这个协议用于约束 AI 如何从公开样本中提炼网文结构规律，避免滑向洗稿、仿写或复用具体桥段。

准确说，我们不是“学习某本爆款怎么写”，而是做：

`公开样本结构观察 + 通用写作机制提炼 + 原创项目适配`

## 可蒸馏内容

允许提炼的是通用结构机制：

- 题材入口：读者为什么愿意点进来。
- 开篇压力：前三章如何建立冲突和期待。
- 情绪曲线：压抑、期待、释放、悬念如何排列。
- 冲突机制：矛盾如何从小局升级到大局。
- 爽点位置：读者何时得到回报。
- 追读钩子：章节末尾如何制造下一章动机。
- 人物关系推进：关系如何通过事件变化，而不是靠解释变化。
- 反转机制：信息差、误判、身份差、资源差如何制造反转。

## 禁止蒸馏内容

禁止提取或复用具体创作资产：

- 具体人设。
- 具体世界观。
- 具体主线骨架。
- 具体桥段。
- 具体台词。
- 具体章节设计。
- 具体书名、人物名、势力名、物品名。
- 某个作者的标志性表达和文风。

如果一个结构卡离开原作品后仍然带有明显原作痕迹，就不能进入 skill 库。

## 结构 Skill 标准格式

```yaml
skill_id: opening_pressure_001
name: 开篇压迫式羞辱
skill_type: structure
status: candidate

use_when:
  - 主角低位开局
  - 需要快速建立反击期待
  - 读者需要在前三章看到明确矛盾

core_mechanism:
  - 公开否定主角价值
  - 让读者看到不公平压力
  - 让主角保留一个暂不暴露的反击条件

reader_effect:
  - 愤怒
  - 期待反击
  - 想知道主角怎么翻局

avoid:
  - 压迫太久导致读者憋闷
  - 主角显得真窝囊
  - 直接照搬退婚、宴会、测试等具体桥段
  - 反派降智到不可信

originality_checks:
  - 是否能换成完全不同题材仍然成立
  - 是否没有复用具体人物、场景、台词
  - 是否只是结构机制，而不是剧情复述

output_contract:
  - 生成一个原创冲突场景
  - 明确主角暂时不能暴露的底牌
  - 结尾留下下一章反击动机
```

## 蒸馏流程

1. 观察样本

只记录结构现象，不记录可直接复用的具体表达。

2. 抽象机制

把具体桥段抽象成通用写作机制。例如“退婚宴羞辱”不能保留，但可以抽象成“公开场合否定主角价值”。

3. 做原创性剥离

删除人名、地名、势力名、物品名、设定名、具体动作链和台词。

4. 写成 skill 卡

只保留 use_when、core_mechanism、reader_effect、avoid、originality_checks。

5. 用新题材测试

如果该 skill 只能在原题材和原桥段中成立，说明抽象不够，不能入库。

6. 人类审核

人类判断是否存在洗稿风险。未审核的 skill 只能是 candidate，不能直接用于正文生成。

## 审核标准

一个结构 skill 可以入库，必须满足：

- 离开原作品仍然成立。
- 不含具体桥段复述。
- 不含标志性设定和台词。
- 能服务多个不同原创项目。
- 有明确的 avoid 条款防止模板化。

## 高风险信号

出现以下情况应立即退回：

- skill 内容像某本书的剧情简介。
- 输出样例与样本作品高度相似。
- 章节顺序、矛盾升级、人物配置与样本一致。
- 只是替换了姓名和场景。
- AI生成时频繁回到同一个爆款桥段。
