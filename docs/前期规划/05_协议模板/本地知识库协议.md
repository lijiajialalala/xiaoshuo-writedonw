# 本地知识库协议

更新时间：2026-05-04

## 目的

本地知识库用于辅助长篇小说写作中的检索、回忆和风险检查。

它解决的是：

- 前文相关片段查找。
- 人物出场和状态查找。
- 伏笔和回收查找。
- 重复桥段和重复表达检查。
- 平台规则和风控资料查找。
- 结构 skill、文笔 skill 和样本观察笔记查找。

## 与主账本的区别

主账本是权威事实源：

- `story_bible.yaml`
- `style_bible.yaml`
- `project_state.json`
- `human_decision_log.yaml`
- `foreshadowing_index.yaml`

本地知识库是检索辅助：

- 章节全文。
- 章节摘要。
- 修改记录。
- 外部规则资料。
- 样本观察笔记。
- skill 文件。

如果知识库检索结果和主账本冲突，以主账本为准。

## 推荐技术栈

第一阶段建议：

- 文件：Markdown/YAML/JSON。
- 数据库：SQLite。
- 全文检索：SQLite FTS5。

暂不优先上复杂向量库，因为第一阶段更需要可控、可查证、可维护。

第二阶段再考虑：

- Chroma。
- LanceDB。
- Qdrant。
- FAISS。

向量检索适合做语义相似和桥段相似，但不适合作为唯一记忆源。

## LLM Wiki 方法的角色

LLM Wiki 是一种知识库构建方法，不等同于飞书本身。可以参考它的“知识库即代码库”“摄入时整理”“持久化知识制品”等原则。

适合放在 LLM Wiki 层的内容：

- source：平台规则、样本观察、调研材料。
- concept：爽点、追读、AI味、洗稿风险、长篇记忆。
- entity：平台、项目、角色、账号、题材。
- skill：结构 skill、文笔 skill、anti-skill、risk skill。
- index：知识目录。
- log：操作日志。

不建议只放在 Wiki 层里的内容：

- 每章写前必须读取的 `context_pack`。
- 机器需要稳定更新的 `project_state`。
- 伏笔状态和人物状态主账本。
- SQLite 检索索引。

原则：Wiki 负责知识沉淀，本地文件和数据库负责执行。

## 写前检索清单

每章生成 `context_pack` 前，至少检索：

- 最近3章摘要。
- 当前出场人物最近3次出场记录。
- 当前冲突相关的旧事件。
- 当前未回收伏笔。
- 最近使用过的爽点、冲突和动作表达。
- 与本章题材/桥段相似的已有章节。
- 平台相关风险规则。

## 写后入库清单

每章定稿后，必须入库：

- 正文路径。
- 章节摘要。
- 关键事件。
- 人物状态变化。
- 新伏笔。
- 已回收伏笔。
- 人类决策记录。
- 审稿结果。
- 重复风险记录。

## 不建议的做法

- 只把全文丢进向量库，不维护 story bible。
- 只保存章节正文，不保存摘要和人物状态。
- 只依赖模型长上下文，不做检索。
- 检索出一堆片段后全部塞给AI。
- 让知识库里的旧草稿覆盖最终定稿事实。
