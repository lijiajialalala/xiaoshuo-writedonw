# 平台待验证清单

## 0. 为什么要有这个清单

公开资料只能解决一部分问题。

真正影响投稿和收益的内容，往往在：

1. 作者后台。
2. 当前合同。
3. 编辑口径。
4. 作者真实反馈。
5. 平台当年活动细则。
6. 违规/断更后的实际处理。

所以本清单不追求一次性答完，而是作为后续调研任务池。

## 5. 作者反馈和案例采集池

这些不是规则来源，只能作为“真实执行线索”。进入运行包前必须用官方规则、后台、合同或编辑口径二次确认。

| 方向 | 要找什么 | 可信度要求 | 用途 |
| --- | --- | --- | --- |
| 番茄全勤/星火到账 | 作者后台截图、结算截图、同一金额被多人印证 | medium以上 | 校准真实新人保底预期和更新成本 |
| 番茄AI低质处罚 | 拒签/下架/停止推荐案例，最好能对应官方通知或站内消息 | medium以上 | 训练正文质量门和异常发文门 |
| 七猫保底报价 | 编辑报价截图、合同条款脱敏截图、同题材多作者反馈 | medium以上 | 判断七猫是否值得作为第二投稿路径 |
| 七猫内投通过率 | 同题材投稿反馈、编辑回复周期、退稿理由 | low到medium | 校准内投包内容和题材选择 |
| 起点当前福利 | 作家助手后台截图、编辑说明、当前合同条款 | medium以上 | 判断起点是否只作为对照样本还是可投路径 |
| 起点收益实操评论 | 作者完结复盘、首订/均订/月收入自述、评论区对补贴到账的反驳或确认 | low到medium | 用来校准收益区间和复核问题，不能直接当规则 |
| 起点内投/直发差异 | 编辑邮箱要求、后台站短时间、拒稿理由、同题材内投反馈 | low到medium | 判断我们是否需要起点内投包，而不是只做番茄直发稿 |
| 起点推荐机制 | 后台公告截图、推荐轮次说明、追读/收藏/上架节点 | medium以上 | 校准起点对照样本的前10章和前20万字拆解颗粒度 |
| AI披露/标识执行 | 平台后台是否出现AI声明/标识功能、合同是否要求披露 | high | 决定正式生产流程是否需要留存AI使用记录 |

## 6. 后续调研方法

### 6.1 来源分级

| 来源 | 用途 |
| --- | --- |
| 官方规则/公告 | 可进入平台机制文档 |
| 作者后台截图 | 可补具体金额和操作路径，但需记录日期 |
| 签约合同样本 | 可补权利义务，但不得公开敏感内容 |
| 编辑口径 | 可作为投稿策略参考，需标注编辑和时间 |
| 作者论坛/社媒反馈 | 只能作为案例和风险线索 |
| 单个作者经验 | 不能直接归纳为平台规则 |

### 6.2 记录格式

每条新资料都按这个格式登记：

```yaml
platform:
topic:
source_type:
source_url_or_note:
sampled_at:
claim:
evidence_summary:
confidence:
decision_impact:
needs_recheck:
```

### 6.3 进入运行包的门槛

平台规则进入运行包前，必须满足：

1. 至少一个官方来源，或两个以上独立高可信来源。
2. 采样日期明确。
3. 能说明它影响哪一个生产决策。
4. 没有和其他来源明显冲突。
5. 如果冲突，必须保留冲突说明，不能强行归纳。
