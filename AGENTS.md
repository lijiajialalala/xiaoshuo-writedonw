# Project Agent Instructions

本仓库服务于“小说生产运行包”：建立一套可复用、可评估、可迭代、可 clean-room 生产长篇小说的系统。

番茄、七猫、起点是当前平台约束和样本来源，不是最终目标。项目目标不是复刻某本书、某个作者、某个爆款开局，也不是只做一本都市高武。

## 2. 最重要的行为规则

用户的例子不是分类结论。

当用户说“比如”“举个例子”“不一定”“我只是说一个可能”时，必须先保留可能性，不能立刻压缩成某个确定层级。

默认响应顺序：

1. 复述用户真正的问题范围。
2. 判断用户是在举例、提假设、做决定、提约束，还是要求执行。
3. 至少保留 2-3 个解释可能。
4. 再说明它可能落在哪些系统位置。
5. 如果需要改文档或模板，说明改哪里，并实际修改。

避免无证据地说：

- “本质上就是……”
- “它不是 A，而是 B。”
- “这个点已经可以归到 X。”
- “所有这类问题都属于……”

优先说：

- “目前更像候选解释，不是定论。”
- “它至少可能落在几层。”
- “这个例子提示我们可能要补一个上游层。”
- “要靠样本对照决定它是不是稳定机制。”

## 3. 当前默认工作流

当前不要无限扩写方法论，也不要急着正式写正文。

```text
P0 平台/赛道/合规假设
-> P0.6 三平台机制与收益研究
-> P1 L0 样本池
-> P2 前 10 章快拆校准批次
-> P2.5 route_candidate 与跨题材基础逻辑候选判断
-> P3 第一卷/事件链深拆
-> P4 内容内核卡
-> P5 clean-room 前 3 章/前 10 章诊断
-> P6 盲审、追读、相似风险回修
-> P7 Story Bible / Style Bible / runtime skill
```

当前重点：

- 不只拆都市高武，也要补跨题材基础叙事逻辑样本。
- P2 三本校准批次先验证 `schemas/chapter_evidence.yaml` 能不能产出 route_candidate。
- 如果 P2 字段不能承载底层逻辑，先回修提取协议。
- 如果 clean-room 前 3 章跑不动，说明承诺、因果、阻力、文风或相似风险至少有一层没拆对。

## 4. 合规和相似风险

硬风险零容忍：平台处罚、法律侵权、签约受阻、合同违约、版权授权冲突。

软风险高优先级：读者觉得像某本书、作者圈质疑、评论区带节奏、项目信誉受损、被认为是 AI 水文或换皮文。

本项目融合的是抽象生成机制，不是多个作品的高识别内容形状。禁止迁移角色身份组合、开局桥段顺序、专属能力、独特怪物/异常规则、标志性组织体系、名场面排列、高识别文风、可被读者一眼认出的爽点路径。

## 5. 仓库边界

```text
.agents/    项目内 skills 和后续 runtime skill
docs/       总控、平台研究、小说蒸馏方法、番茄专项、前期归档
research/   样本池、榜单表、拆解表、阶段研究记录
schemas/    样本、章节证据、内容内核、盲审、风险循环模板
data/       可入库小样本和安全夹具，不放大体量原文
scripts/    自动化抓取、清洗、校验脚本
src/        后续可执行流水线代码
tests/      后续回归测试和质量验证
```

- 不提交大体量原始小说正文。
- 不提交受版权保护的整章、整卷、整本内容。
- 样本正文默认放仓库外部，只记录来源、路径、章节范围和证据摘要。
- 新增顶层目录前，先确认是否能放入现有目录。

## 6. 验证

文档或模板修改后至少运行：

```powershell
git diff --check
git status --short
```

提交前检查：

- 是否误放大体量原文。
- 是否新增未经证据支持的绝对结论。
- 是否把单本书特色写成通用规律。
- 是否遗漏合规和相似风险。
