# Project Agent Instructions

本仓库服务于“小说生产运行包”：建立一套可复用、可评估、可迭代、可 clean-room 生产长篇小说的系统。

番茄、七猫、起点是当前平台约束和样本来源，不是最终目标。项目目标不是复刻某本书、某个作者、某个爆款开局，也不是只做一本都市高武。

## 0. 指令文件定位

公开优秀 `AGENTS.md` / `CLAUDE.md` 的共同点是：项目目标清楚、do/don't 具体、命令和验证明确、不过度堆长篇教程。

因此本文件只放 agent 必须长期遵守的行为规则；详细方法见：

- `README.md`
- `docs/README.md`
- `docs/00_总控/00_当前总控路线图.md`

- `docs/20_小说蒸馏方法/03_小说提取归纳总模型.md`
- `docs/20_小说蒸馏方法/04_P2小说证据提取器协议.md`

## 1. 第一原则

任何小说蒸馏、平台研究、样本拆解、route 归纳、文风建模、clean-room 试写，都服务于：

```text
从多本样本中提取可迁移的抽象生成机制，
经过平台约束、合规边界、叙事机制、文风渲染和 clean-room 验证，
形成我们自己的 Story Bible / Style Bible / 章节规划器 / 场景生成器。
```

禁止把目标缩窄成：

- 复刻某本书或某个作者。
- 写一篇“像番茄”的开局。
- 只研究都市高武。
- 把通过盲审当成最终生产许可。
- 把平台短期偏好误判成小说通用能力。

## 2. 用户例子不是分类结论

这是本仓库最重要的 agent 行为规则。

当用户说“比如”“举个例子”“不一定”“我只是说一个可能”时，必须当作非穷尽例子，不能立刻压缩成确定层级。

错误做法：

```text
你说的不是文风，也不是题材 route，而是“主角为什么必须持续行动”的基础叙事逻辑层。
```

正确做法：

```text
先保留可能性，不急着定层。
这个例子可能属于主角发动机，也可能属于压力逻辑、代价逻辑、长期承诺逻辑，甚至说明系统缺一层“跨题材基础叙事逻辑库”。
```

默认响应顺序：

1. 复述用户真正的问题范围。
2. 判断用户是在举例、提假设、做决定、提约束，还是要求执行。
3. 至少保留 2-3 个解释可能。
4. 再说明它可能落在哪些系统位置。
5. 如果需要改文档或模板，说明改哪里，并实际修改。

## 3. 反过早收缩

遇到创作方法、拆书策略、平台判断、文风判断时，不能直接给单一结论。

先问：

```text
这是不是例子，而不是结论？
它可能属于哪些层？
它是否说明当前系统缺少新层？
如果我现在定层，会不会误导后续提取字段？
这个结论会改变哪一个生产动作？
```

除非已有样本证据，否则避免说：

- “本质上就是……”
- “它不是 A，而是 B。”
- “这个点已经可以归到 X。”
- “所有这类问题都属于……”

优先说：

- “目前更像候选解释，不是定论。”
- “它至少可能落在几层。”
- “这个例子提示我们可能要补一个上游层。”
- “要靠样本对照决定它是不是稳定机制。”

## 4. 必须使用的问题透镜

用户提出任何小说创作或系统设计想法时，至少快速过一遍这些透镜：

- 跨题材基础叙事逻辑：这是不是所有长篇都需要的“油盐酱醋”？
- 平台约束：这是番茄/七猫/起点机制导致的，还是小说通用能力？
- 读者承诺：读者追看的主回报是什么？
- 主角发动机：主角为什么必须行动，为什么不能停？
- 叙事因果：上一幕如何逼出下一幕？
- 阻力系统：对手、规则、资源、误判、组织、关系如何反咬？
- 揭示/翻牌：信息如何藏、误导、释放，并改写局势？
- 能力/设定功能：设定是否制造限制、代价、线索、反制和新问题？
- 文风/叙述渲染：叙述距离、对白、心理、动作、信息密度如何服务追读？
- 合规/相似风险：会不会落入硬风险或软风险？
- 执行流程缺口：当前模板、字段、验证门是否能承载这个问题？
- agent 行为问题：是不是 agent 自己过早归纳、过度自信或漏问了 why？

## 5. 当前核心分层

小说不是“更长的文案”。默认按这些层观察，但不要机械套层：

1. 平台/商业约束层。
2. 跨题材基础叙事逻辑层。
3. 题材/赛道层。
4. route 生成路线层。
5. 单书读者承诺层。
6. 卷/副本/事件链层。
7. 章节推进层。
8. 场景因果层。
9. 人物/关系/组织层。
10. 能力/设定功能层。
11. 揭示/翻牌层。
12. 叙事身体层。
13. 文风/语言渲染层。
14. 合规与相似风险层。

## 6. 跨题材基础叙事逻辑库

本项目必须建立“跨题材基础叙事逻辑库”，从不同风格、不同平台、不同榜单小说中抽取长篇持续可读的底层机制。

这一层关注“做饭的油盐酱醋”，不是具体菜系。

候选基础逻辑包括但不限于：

- 缺口逻辑：主角缺什么，读者等什么被补上。
- 压力逻辑：如果主角不行动，会失去什么。
- 代价逻辑：每次得到都要付出什么，代价如何升级。
- 资源循环：资源如何被消耗和再获得。
- 信息差逻辑：读者、主角、对手分别知道什么。
- 关系牵连：一个行动如何改变关系网。
- 规则约束：世界、组织、能力、平台或副本规则如何限制行动。
- 兑现延迟：承诺如何分期兑现，而不是一次性用完。
- 状态变化：每章/每事件结束后局势有没有真实变化。
- 后果逻辑：赢了以后为什么不是结束，而是引出更大问题。
- 反咬逻辑：主角的胜利、能力或选择如何制造新风险。
- 读者追问逻辑：章末留下的不是空钩子，而是明确的下一步好奇。

抽取顺序优先为：

```text
跨题材基础逻辑
-> 平台差异对照
-> 题材 route 差异
-> 单书机制
-> 文风和渲染器
```

不能先锁死都市高武，再把都市高武里的机制误判成小说通用规律。

## 7. route 不是题材标签

题材是“写什么领域”，route 是“这类书如何生成追读”。

判断 route 必须回答：

- 主承诺是什么？
- 主角靠什么赢？
- 阻力如何反咬？
- 爽点/悬疑/成长如何组合兑现？
- 信息如何翻牌？
- 能力体系如何持续制造剧情，而不是只比大小？
- 这条 route 对新人是否可执行？
- 这条 route 能否支撑 30-100 万字？

## 8. 提取和归纳纪律

提取不是读后感，归纳不是平均混合。

提取必须回答：

- 原文证据在哪里？
- 发生在哪一章/哪一段/哪一个事件？
- 这条证据承担什么功能？
- 它是单书特色、题材套路、平台约束，还是跨题材基础逻辑候选？

归纳必须回答：

- 多条证据共同说明什么生成机制？
- 有没有反例？
- 反例是否推翻、限制或修正该机制？
- 这个机制能迁移到我们项目吗？
- 迁移时哪些高识别内容不能带走？
- 它会改变哪一个生产动作？

如果结论不能改变生产动作，不要放进 runtime，只能放进观察记录。

## 9. critical-thinking-mode 的项目化执行

本仓库已保存：

```text
.agents/skills/critical-thinking-mode/SKILL.md
```

关键结论必须带最小自问自答：

```text
Q1：我现在是不是把例子当成结论了？
Q2：这个判断有哪些替代解释？
Q3：需要什么样本证据才能支持它？
Q4：什么证据会推翻它？
Q5：如果它成立，会改变哪个提取字段、归纳模板或生产动作？
```

触发场景：

- 用户指出 agent 理解错了。
- 新增系统层级。
- 宣布某个策略可行。
- 从样本共性升级成规则。
- 从规则升级成 runtime。
- clean-room 试写通过或失败。
- 涉及合规、洗稿、平台处罚、合同、收益判断。

## 10. 合规和相似风险

硬风险零容忍：

- 平台处罚。
- 法律侵权。
- 签约受阻。
- 合同违约。
- 版权授权冲突。

软风险高优先级：

- 读者觉得像某本书。
- 作者圈质疑。
- 评论区带节奏。
- 项目信誉受损。
- 被认为是 AI 水文或换皮文。

本项目融合的是抽象生成机制，不是多个作品的高识别内容形状。

禁止迁移角色身份组合、开局桥段顺序、专属能力、独特怪物/异常规则、标志性组织体系、名场面排列、高识别文风、可被读者一眼认出的爽点路径。

## 11. 置信度纪律

不能说“100% 有信心”，除非经过真实样本验证、clean-room 迁移和追读/盲审反馈。

必须区分：

- 策略结构置信：流程设计逻辑上自洽。
- 执行模板置信：模板字段能被稳定填写。
- 样本归纳事实置信：多本样本证据支持该规律。
- clean-room 生产置信：该规律能指导非贴皮文本。
- 平台适配事实置信：真实平台反馈支持该打法。

没有样本和验证时，只能说“结构上有理由成立”，不能说“事实成立”。

## 12. 当前工作流

当前不要无限扩写方法论，也不要急着正式写正文。

默认路线：

```text
P0 平台/赛道/合规假设
-> P0.6 三平台机制与收益研究
-> P1 L0 样本池
-> P2 前 10 章快拆校准批次
-> P2.5 route_candidate 与跨题材基础逻辑候选判断
-> P3 第一卷/事件链深拆
-> P4 内容内核卡
-> P5 clean-room 前 3 章/前 10 章诊断
-> P6 盲审、追读、相似风险回修
-> P7 Story Bible / Style Bible / runtime skill
```

当前重点：

- 不只拆都市高武，也要补跨题材基础逻辑样本。
- P2 三本校准批次先验证 `schemas/chapter_evidence.yaml` 能不能产出 route_candidate。
- 如果 P2 字段不能承载底层逻辑，先回修提取协议。
- 如果 clean-room 前 3 章跑不动，说明承诺、因果、阻力、文风或相似风险至少有一层没拆对。

## 13. 仓库边界和验证

仓库结构：

```text
.agents/    项目内 skills 和后续 runtime skill
docs/       总控、平台研究、小说蒸馏方法、番茄专项、前期归档
research/   样本池、榜单表、拆解表、阶段研究记录
schemas/    样本、章节证据、内容内核、盲审、风险循环模板
data/       可入库小样本和安全夹具，不放大体量原文
scripts/    自动化抓取、清洗、校验脚本
src/        后续可执行流水线代码
tests/      后续回归测试和质量验证
```

边界：

- 不提交大体量原始小说正文。
- 不提交受版权保护的整章、整卷、整本内容。
- 样本正文默认放仓库外部，只记录来源、路径、章节范围和证据摘要。
- 新增顶层目录前，先确认是否能放入现有目录。

文档或模板修改后至少运行：

```powershell
git diff --check
git status --short
```

提交前检查：

- 是否误放大体量原文。
- 是否新增未经证据支持的绝对结论。
- 是否把单本书特色写成通用规律。
- 是否遗漏合规和相似风险。

## 14. 最重要的默认姿态

遇到用户提出的新想法，先不要急着“安放到系统里”。

先问：

```text
这是不是在提醒我们：当前系统缺一层？
```

再问：

```text
这个想法如果成立，我们的提取字段、归纳模板、验证门、生产动作分别要怎么变？
```

只有回答完这两个问题，才允许下结论。
